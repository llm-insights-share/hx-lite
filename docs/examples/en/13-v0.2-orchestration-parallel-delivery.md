## Background

After upgrading to **v0.2**, the inventory-service team adds **execution orchestration** on top of the HarnessX control plane: parallel agent tasks per change, isolated worktree trial runs, and line-level code review feedback fed directly into the apply loop.

This scenario covers v0.2 orchestration capabilities:

| Capability | CLI | Purpose |
| --- | --- | --- |
| GitHub Issue scaffold | `hx change create --from-issue` | Open a change from an Issue |
| Parallel apply | `hx apply --parallel` + `@group=` | Run independent tasks concurrently |
| Best-of-N fan-out | `hx apply --fan-out` | Try N worktrees, pick best verify score |
| Worktree runtime | `hx runtime worktree` | Create/manage isolated environments |
| Diff annotation feedback | `hx review import` | Line-level review → `HX_FIX_HINTS` |
| Guide behavior evals | `hx eval guides` | Validate Context Pack quality before release |
| Change notifications | `hx notify` | Long-run completion / approval reminders |

Roles:

- **Chen** (backend): primary dev, runs CLI agents for apply (terminal / Cursor);
- **Liu** (PM): files GitHub Issues that seed changes;
- **Zhao** (reviewer): exports line comments from code review into JSON for HarnessX.

## Steps

### 1. Create a change from a GitHub Issue (v0.2)

Liu filed Issue #87 “Auto-release inventory reservation on timeout”:

```console
$ hx change create reserve-timeout --from-issue https://github.com/acme/inventory-service/issues/87
Created change "reserve-timeout" from issue #87
  proposal: harnessX/changes/reserve-timeout/proposal.md
  delta: harnessX/changes/reserve-timeout/specs/inventory/spec.md
```

HarnessX calls the GitHub API, prefills the **Source Issue** section in `proposal.md`, and infers `touchedDomains` from Issue labels (defaults to `core` if none). Chen can override domains:

```console
$ hx change create reserve-timeout --from-issue https://github.com/acme/inventory-service/issues/87 --domains inventory,reservation
```

Then follow the usual propose→design flow (scenario 02) until **planned**.

### 2. Annotate parallel groups and dependencies in tasks.md (v0.2)

After `hx plan reserve-timeout`, Chen tags independent test tasks with `@group=` and impl tasks with `@depends=`:

```markdown
# Tasks: reserve-timeout

- [ ] 01a [test] (inventory / Requirement: Reservation timeout release) Write failing test(s)... @group=tests-a
- [ ] 02a [test] (inventory / Requirement: Reservation renewal) Write failing test(s)... @group=tests-a
- [ ] 01b [impl] (inventory / Requirement: Reservation timeout release) Implement... @depends=01a
- [ ] 02b [impl] (inventory / Requirement: Reservation renewal) Implement... @depends=02a
```

Semantics:

- `@group=tests-a`: 01a and 02a have **no dependency on each other** → eligible for concurrent execution;
- Tasks **without** `@group` always run **serially** (one at a time);
- `@depends=01a`: 01b waits until 01a completes (impl always after its test task).

### 3. Parallel apply (v0.2)

In the main repository worktree:

```console
$ hx apply reserve-timeout --parallel 2 --runner 'codex exec --prompt "$HX_TASK_TITLE"'
completed tasks: 01a, 02a, 01b, 02b; remaining: 0
```

Scheduler behavior:

1. Batch 1: ready + same group → `[01a, 02a]` concurrently (up to 2);
2. Batch 2: `01b` (depends on done 01a);
3. Batch 3: `02b` (depends on done 02a).

Each task still runs the profile **fast suite** afterward; failures feed `$HX_FIX_HINTS` for self-correction (same as v0.1, default `--max-retries 3`).

### 4. Best-of-N when agents are flaky (v0.2)

Reservation logic is tricky; a single apply sometimes fails verify. Chen uses fan-out:

```console
$ hx apply reserve-timeout --fan-out 3 --runner 'claude -p "Implement task: $HX_TASK_TITLE"'
fan-out: 3 worktrees for reserve-timeout
  fanout-1: score=1000 verify=pass blockers=0
  fanout-2: score=1000 verify=pass blockers=0
  fanout-3: score=98 verify=fail blockers=2
selected: fanout-1 (/path/to/../hx-reserve-timeout-fanout-1)
```

Results are stored in `harnessX/changes/reserve-timeout/runtime.yaml` under `fanOut.candidates` and `selected`.

**Merge rule unchanged**: confirm diff in the selected worktree and merge to main, or cherry-pick. Clean up losers:

```console
$ hx runtime worktree remove reserve-timeout --slot fanout-3
worktree removed
```

Manual worktrees without fan-out:

```console
$ hx runtime worktree create reserve-timeout --slot wt-a
worktree wt-a: /path/to/../hx-reserve-timeout-wt-a (branch cursor/hx-reserve-timeout-wt-a-5afb)
```

Run `hx apply reserve-timeout --runner '...'` inside `wt-a`, then `hx verify reserve-timeout` — green verify remains the merge bar.

### 5. Line-level diff annotations → fix_hints (v0.2)

Zhao exports PR review comments as `reviews/reserve-timeout.json` (HarnessX native JSON):

```json
[
  {
    "file": "src/reservation/timeout.ts",
    "line": 48,
    "severity": "critical",
    "comment": "TTL must come from config, not hardcoded 300"
  }
]
```

Import:

```console
$ hx review import reserve-timeout reviews/reserve-timeout.json
imported 1 annotation(s)
  hint: [CRITICAL] src/reservation/timeout.ts:48: TTL must come from config, not hardcoded 300
```

Continuing apply merges these hints into **`HX_FIX_HINTS`** alongside sensor output. When fixed:

```console
$ hx review resolve reserve-timeout import-1
resolved import-1
```

### 6. Guide behavior evals before promoting assets (v0.2)

After editing `assets/workflows/dev/apply.md`:

```console
$ hx eval guides reserve-timeout
propose-has-template: PASS
apply-no-proposal-noise: PASS
spec-has-spec-writing: PASS
plan-phase-scoped: PASS
all guide evals passed
```

Cases ship in `packages/core/evals/guide-behavior.json`; teams can add `harnessX/evals/team-cases.json`.

### 7. Notifications for long runs (v0.2)

```console
$ hx notify reserve-timeout --once
watching reserve-timeout (implementing, tasks 2/4)
[hx watch] reserve-timeout needs_approval: Spec gate needs human approval before plan

$ export HX_WATCH_WEBHOOK=https://hooks.example.com/harnessx
$ hx notify reserve-timeout --interval 60000
```

Event kinds: `needs_approval`, `tasks_complete`, `review_pending`, `status_change`.

> **Naming**: `hx watch` remains the file-save sensor daemon (scenario 10). Change polling is **`hx notify`**.

### 8. Close out: verify → archive

```console
$ hx verify reserve-timeout
VERIFIED

$ hx archive reserve-timeout
Archived to harnessX/archive/2026-07-05-reserve-timeout
Merged capabilities: inventory
```

## Relationship to scenario 09

| Dimension | Scenario 09 | This scenario (v0.2) |
| --- | --- | --- |
| Focus | Multi-**tool** adapter compile (Cursor/Trae/Qoder/Claude) | Multi-**agent** / worktree **parallel runs** |
| Isolation | Qoder Quest manual `git worktree add` | First-class `hx runtime worktree` / `--fan-out` |
| Human feedback | PR review → Steering harvest (batch, post-hoc) | `hx review import` → immediate fix_hints (in apply loop) |
| Entry | `hx adapter sync` | `--from-issue` + `hx notify` |

Scenario 09 prevents rule drift; scenario 13 orchestrates **how** agents run and **how** review comments loop back.

## Key mechanisms

- **Orchestration does not replace gates**: `--parallel` / `--fan-out` only change **how** apply runs; `hx verify` remains the acceptance bar.
- **Merged fix_hints**: sensor reports + review annotations share `$HX_FIX_HINTS`.
- **runtime.yaml vs meta.yaml**: fan-out candidates and worktree paths are operational state, not gate ledger data.
- **Fan-out scoring**: verify pass ≈ score 1000 minus blockers; tie-break is first candidate — human can override before merge.
- **Issue scaffold is a starting point**: `--from-issue` seeds proposal + delta; spec quality still depends on gates and human approval (FR-012).
