# Scenario 16: Initialize from Hub blueprint + asset sync merge (v0.3)

## Background

The platform team uses **Harness Hub** as the org asset center:

```
harness-hub/
├── packages/          # Skills, templates, rubrics
├── bundles/           # Topology bundles (api-service, frontend-2c, …)
├── blueprints/        # Delivery blueprints (profile + hub_deps)
└── evals/             # Golden-repo validation sets
```

A new **member API** service is onboarded by backend lead **Zhou**. Platform policy: **no copy-paste** of Skills across repos — consume from Hub; local customizations go through `overrides` + `hub sync --apply` three-way merge.

Roles: **Zhao** (platform Hub maintainer), **Zhou** (member-api owner).

## Steps

### 1. Platform team: seed Hub and search assets

```console
$ hx hub seed ./harness-hub
Seeded ./harness-hub with 8 package(s):
  api-conventions@1.0.0
  prd-writing@1.0.0
  ...

$ hx hub search prd --hub ./harness-hub
package	prd-writing@1.0.0	guide.skill

$ hx hub search --hub ./harness-hub --category blueprint
blueprint	enterprise-delivery@1.0.0	harness.blueprint	enterprise-delivery

$ hx hub search --hub ./harness-hub --index
wrote ./harness-hub/index.json
```

### 2. Application repo: one-shot init from Hub bundle

```console
$ cd member-api
$ hx init --from-hub api-service@1.0.0 --hub ../harness-hub --adapter cursor
Initialized from hub api-service@1.0.0 → /work/member-api/harnessX
  + hub bundle api-service@1.0.0
  ...

$ hx bundle list --hub ../harness-hub
api-service@1.0.0	(hub)
frontend-2c@1.0.0	(hub)

$ hx hooks install && hx ci init && hx adapter sync
```

`config.yaml` records `hub: ../harness-hub`; `harness.lock` pins resolved asset hashes.

### 3. Optional: enterprise delivery blueprint

For the full enterprise path (requirements → PRD → prototype → UAT):

```console
$ hx init --from-hub enterprise-delivery@1.0.0 --hub ../harness-hub
```

The blueprint sets `profile: enterprise`, installs Hub deps (`prd-writing`, `prototype-wireframe`, `uat-checklist`, …), and writes `harnessX/blueprint.yaml`.

### 4. Hub upgrades and three-way merge

```console
$ hx hub sync --hub ../harness-hub
api-conventions	installed 1.0.0	latest 1.1.0	update-available

$ hx hub sync --hub ../harness-hub --apply
api-conventions	updated → 1.1.0

$ hx lock write && hx lock verify
harness.lock verified
```

With local overrides on the cache, state becomes `update-and-local-changes`. On conflict:

```console
$ hx hub sync --hub ../harness-hub --apply --force
api-conventions	conflict → 1.1.0 conflicts: SKILL.md
```

### 5. Steering → Hub closed-loop publish

```console
$ hx steer publish harnessX/assets/guides/idempotency-keys \
    --hub ../harness-hub --by zhou.dev
eval: PASS (5 checks)
published to ../harness-hub/packages/idempotency-keys/0.1.0 (hub review pending)
```

### 6. Codex CLI and tier compensation

```console
$ hx adapter sync --targets codex,generic
$ cat .harnessx-adapter-tier
2

$ hx apply add-member-tier --runner 'codex exec --prompt "$HX_TASK_TITLE"'
```

Tier 2 augments gate suites and escalates warn→block where configured.

## Key mechanisms

| Capability | What it does |
| --- | --- |
| `init --from-hub` | Init from bundle, blueprint, or single package + lock |
| `blueprint.yaml` | Reusable delivery path (profile + hub_deps) |
| `hub sync --apply` | Baseline / local override / upstream three-way merge |
| Tier → Gate compensation | Stronger L3 checks for weaker IDE adapters |
| `steer publish` | metrics → eval → hub promote with evidence chain |
