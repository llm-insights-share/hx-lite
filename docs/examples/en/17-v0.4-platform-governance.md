## Steps

### 1. Design â€” `prototype-complete` gate

When UI is in scope, the enterprise `design-enterprise` suite includes `prototype-complete`:


```console

```

### 2. Verify â€” `uat-complete` gate

Before verify, Chen copies the Hub UAT template:

```console

```

Create `harnessX/changes/member-badge/uat-checklist.md`:

```markdown

| Scenario ID | Steps | Expected | Pass | Tester | Date |
```

```console
$ hx gate check member-badge --phase verify

```

### 3. Unified `drift` sensor

The `drift` builtin aggregates:

| Rule | Source |
| --- | --- |
| `spec-code-drift` | Archived specs vs test references (`hx sync`) |
| `design-code-drift` | Missing `code_hints` from `delivery-trace.yaml` |
| `adapter-drift` | Hand-edited `.cursor/rules/harnessx.mdc`, etc. |

```console
$ hx sync
[scenario-without-test] Scenario: badge-empty-state
  â†’ Spec declares "badge-empty-state" but no test references it...

$ hx adapter drift
DRIFT .cursor/rules/harnessx.mdc: manually-edited
$ hx adapter sync
```

### 4. `integration-smoke` slot

The api-service bundle adds `integration-smoke` to verification. With `package.json`:

```json
"scripts": { "test:integration": "vitest run --config vitest.integration.config.ts" }
```

the sensor runs on verify. Without the script it **skips** with an info finding.

### 5. Cross-repo Harness Coverage


### 6. Delivery dashboard `hx view` (v0.4)

```console

```

Includes metric cards, phase funnel, change table, and asset effectiveness (top assets by runs).
