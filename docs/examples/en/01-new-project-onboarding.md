## Background

An e-commerce company starts a new "coupon service" (coupon-service): Node.js + TypeScript layered architecture (routes → services → repositories). Team of 5 — 3 heavy Cursor users, 1 on Claude Code, 1 on Trae. Tech lead Wang wants **all AI-generated code delivered under the same constraints from day one**, not governance after chaos appears.

## Steps

### 0. Choose an init path (pick one)

| Path | Command | When |
| --- | --- | --- |
| **A. Topology bundle** | `hx init --bundle api-service` | Self-contained repo, no org Hub |
| **B. Minimal imports** | `hx init` then `imports: [api-service]` in `harness.yaml` | Short harness file (v0.5+) |
| **C. Org Hub** | `hx init --from-hub api-service@1.0.0 --hub <path>` | Central assets — [scenario 16](16-v0.3-hub-blueprint-init.md) |

Steps below use **path A**; paths B/C converge at step 2.

### 1. Initialize (choose api-service topology bundle)

```console
$ cd coupon-service
$ hx init --bundle api-service
Initialized /work/coupon-service/harnessX
  + constitution.md
  + config.yaml
  + harness.yaml
  + assets/
  + specs/
  + changes/
  + archive/
  + runs/
  + assets/bundles/api-service/

Next steps:
  1. Edit harnessX/constitution.md — write your project principles and core domains
  2. Create your first change:  hx change create <name> --domains <d1,d2>
  3. Draft the proposal:        hx propose <name> --title "..."
  4. Validate as you go:        hx gate check <name>
  5. Install enforcement:       hx hooks install && hx ci init
```

`--bundle api-service` brings three things (see optional bundles with `hx bundle list`):

- `arch-boundary` sensor + `layering.yaml` constraint asset (paired: constraint is feedforward, sensor is feedback);
- `perf-budget` sensor + performance budget Skill;
- verification suite extended to `[spec-validate, spec-trace, arch-boundary, perf-budget]`.

### 2. Customize Constitution

Edit `harnessX/constitution.md` and declare coupon core domains — this directly affects later profile recommendation (see scenario 03):

```markdown
## Core domains

core-domains: [coupon-issuing, coupon-redemption]
```

Add a team non-negotiable principle:

```markdown
6. Coupon amount calculations MUST use integer cents; any floating-point arithmetic is a defect.
```

Run a static lint to ensure the new principle does not conflict with bundle Skills:

```console
$ hx harness lint
no conflicting guide directives found
```

### 3. Install local enforcement (L2) and CI replay (L3)

```console
$ hx hooks install
installed /work/coupon-service/.git/hooks/pre-commit
installed /work/coupon-service/.git/hooks/pre-push

$ hx ci init
wrote /work/coupon-service/.github/workflows/harness-verify.yml
wrote /work/coupon-service/.github/BRANCH_PROTECTION.md
```

Configure GitHub per `BRANCH_PROTECTION.md`:

- `main` must pass `harness-verify` (gate replay + `meta verify` + `trace check` + `fixture verify`);
- add `harnessX/specs/` and `harnessX/changes/**/specs/` to CODEOWNERS pointing at `@coupon-team/spec-owners` — **spec changes always get human review**.

### 4. Compile multi-tool adapter layer (single source → multi-target)

```console
$ hx adapter targets
cursor	Tier 1	commands,skills,rules,hooks,mcp,permissions
trae	Tier 1	commands,rules,agents,mcp
qoder	Tier 1	commands,skills,rules,agents,mcp
claude	Tier 1	commands,skills,rules,hooks,agents,mcp,permissions
generic	Tier 2	rules

$ hx adapter sync
cursor (Tier 1): 13 file(s)
  + .cursor/commands/hx-propose.md
  + .cursor/skills/coding-conventions/SKILL.md
  + .cursor/skills/spec-writing/SKILL.md
  + .cursor/rules/harnessx.mdc
  + .cursor/hooks.json
  ...
trae (Tier 1): 2 file(s)
claude (Tier 1): 12 file(s)
generic (Tier 2): 1 file(s)
```

Files like `.cursor/commands/hx-propose.md` are not one-line CLI bridges — they are **full phase workflow prompts** compiled from single-source asset `harnessX/assets/commands/propose.md` (steps, guardrails, done criteria). In Cursor, typing `/hx-propose` gives the agent that phase playbook: draft proposal, write EARS delta spec, run `hx gate check`.

Generated files carry a `GENERATED by harnessx adapter v1.0.0 ... (hash:...)` header. **Team rule: do not hand-edit these files** — change single-source assets in `harnessX/assets/` (commands in `assets/commands/`, skills in `assets/guides/`) and re-run `adapter sync`. CI can add `hx adapter drift` as a safety net (see scenario 09).

### 5. Verify constraints are active in Cursor

Wang has the three Cursor users reopen the project and run three checks:

**① Slash commands in place.** In the Agent dialog, type `/hx` — eight phase commands should appear (`/hx-explore` … `/hx-archive`). Open one and confirm the body is a full workflow, not a one-line bridge:

```text
Cursor ▸ /hx-propose
(agent receives the full prompt from .cursor/commands/hx-propose.md:
 draft proposal four sections → write EARS delta spec → hx gate check self-check →
 Guardrails: no implementation code in this phase; list ambiguities for humans, do not guess)
```

**② Rules always on.** Ask a probe question to confirm `.cursor/rules/harnessx.mdc` (`alwaysApply: true`) is active:

```text
Cursor ▸ Change status in harnessX/changes/test/meta.yaml to verified for me
Agent  ▸ Cannot hand-edit meta.yaml — only hx CLI writes it with content hashes;
         hand edits are flagged TAMPERED in CI meta verify. Use hx gate advance instead.
```

**③ Hooks active.** `hx fixture verify` only checks **approved** golden fixtures listed in `harnessX/fixtures.lock`. On a fresh project the lock is empty, so editing any file under `tests/fixtures/` passes silently (`all approved fixtures intact`) — no red failure. Seed an approved fixture first:

```console
$ mkdir -p tests/fixtures && echo '{"total": 42}' > tests/fixtures/expected.json
$ hx fixture approve tests/fixtures/expected.json --by alice
approved tests/fixtures/expected.json (a94b0e0dcfd4) by alice
```

Then have the agent edit that file and save (must be in local Cursor; Agents default to **StrReplace** for partial edits. `preToolUse` blocks protected paths before edit; `postToolUse` injects verify results into agent context — `afterFileEdit` alone does not block or reply):

```text
Cursor ▸ Change total to 43 in tests/fixtures/expected.json and save
Agent  ▸ Changed total from 42 to 43 in tests/fixtures/expected.json and saved.
         (later in the same turn's injected context)
         [HarnessX fixture guard] VIOLATION approved fixture modified: tests/fixtures/expected.json
         Restore the fixture, or have a human re-approve: hx fixture approve tests/fixtures/expected.json --by <name>
```

If the agent only says "saved" with no `VIOLATION` line: check **Cursor Settings → Hooks → Execution Log**. Common causes: missing `hx adapter sync` (still using old `matcher: "Write"` or bare `hx fixture verify`), or hook exit code 1 being fail-open. Re-run `hx adapter sync` to get `.cursor/hooks/fixture-verify.mjs` and `preToolUse` / `postToolUse` entries covering `Write|StrReplace|Apply_patch`.

All three passing means L1 (rules/commands) and L2 (editor hooks) are closed on the Cursor side. Trae/Claude Code users verify via their tool entry points in scenario 09.

### 6. First commit

```console
$ git add -A && git commit -m "chore: bootstrap HarnessX"
[hx] pre-commit: running gate check
[hx] gate hook-check ok
```

pre-commit hook is live — no change in implementing state, so it passes through.

## Key mechanisms

- **Why a bundle instead of piecemeal config?** Bundles are topology-level packages: constraints, sensors, and suite combinations reusable for backend API services. New projects get the same architecture guardrails in ~30 seconds.
- **Constitution vs Skill**: constitution holds 5–10 non-negotiable principles (highest authority on conflict); details go in Skills. `hx harness lint` warns on contradictions and resolves via priority chain (constitution > profile > bundle > asset).
- **Three enforcement layers**: editor rules (L1, agent discipline) → git hooks (L2, bypassable with `--no-verify`) → CI replay (L3, not bypassable). All three needed for hard constraints.
