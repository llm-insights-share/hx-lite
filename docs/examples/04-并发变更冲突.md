# 场景 04：两个团队同时修改同一 capability
| | |
| --- | --- |
| **旅程** | 日常交付 |
| **适用角色** | 多团队协作者 |
| **前置** | 场景 01,02 |
| **关联** | 见 [场景选择指南](00-场景选择指南.md) |

## 背景

库存中心（inventory-service）本周有两条并行需求，都涉及 `stock-reservation`（库存预占）能力：

- **A 团队（吴工）**：`reservation-ttl` —— 预占单增加 TTL，超时自动释放；
- **B 团队（郑工）**：`bulk-reserve` —— 支持批量预占接口。

两条 change 并行开发，最终都要把 delta 合并进同一份 `harnessX/specs/stock-reservation/spec.md`。这是 AI 高吞吐交付下最常见的冲突形态：**规格层面的并发写**。

## 操作步骤

### 1. 创建时即感知重叠

吴工先建：

```console
$ hx change create reservation-ttl --domains stock-reservation
Created change "reservation-ttl" (profile: standard, domains: stock-reservation)
```

郑工后建，系统立刻提示域重叠：

```console
$ hx change create bulk-reserve --domains stock-reservation,api-gateway
Created change "bulk-reserve" (profile: standard, domains: stock-reservation, api-gateway)
WARNING: overlaps with active change "reservation-ttl" on domains: stock-reservation
```

这只是**告警不阻断**——并行是常态，但两位负责人此刻就该对齐："你动哪些 Requirement，我动哪些"。郑工看了吴工的 delta，发现双方都要改既有的 `Requirement: 预占创建`（吴加 TTL 字段语义，郑加批量语义）。约定：吴工先合，郑工基于合并结果 rebase。

### 2. A 团队先完成，正常 archive

吴工的 delta（节选）：

```markdown
## MODIFIED Requirements

### Requirement: 预占创建
WHEN 收到预占请求, THE SYSTEM SHALL 创建带 TTL（默认 900 秒）的预占单并返回预占号。

#### Scenario: reservation expires after ttl
- GIVEN 一笔 TTL 900 秒的预占
- WHEN 900 秒无确认
- THEN 预占释放，库存回补
```

走完全流程后 `hx archive reservation-ttl`，主规格中"预占创建"已带 TTL 语义。

### 3. B 团队 archive 前的 rebase 检查

郑工的 delta 是基于**旧版**主规格写的 `MODIFIED Requirements → 预占创建`。在 verify 之后、archive 之前，先做 rebase 检查（archive 内部也会强制做同样的检查，提前跑可以早发现）：

```console
$ hx rebase check bulk-reserve
deltas apply cleanly against current specs
```

这次恰好干净——因为 MODIFIED 是"整体替换"语义，郑工的版本会覆盖吴工刚合入的文本，**这正是危险所在**：静默覆盖会把 TTL 语义冲掉。真正的冲突暴露发生在另一种情形：假设吴工在自己的 change 里把该需求**改名**为"预占创建与过期"，郑工的 MODIFIED 就找不到目标了：

```console
$ hx rebase check bulk-reserve
CONFLICT stock-reservation/"预占创建" (MODIFIED): not found in base spec (concurrent change may have removed/renamed it)
  → Requirement "预占创建" no longer exists in stock-reservation/spec.md — a concurrent change likely renamed or removed it. Re-read the current spec and rewrite your MODIFIED entry against it.

$ hx archive bulk-reserve
BLOCKED: stock-reservation/"预占创建" (MODIFIED): not found in base spec (concurrent change may have removed/renamed it). Rebase your delta against current specs/ and retry.
```

### 4. 解决冲突：基于当前主规格重写 delta

郑工的正确动作不是改冲突报告，而是**重读当前主规格**（`harnessX/specs/stock-reservation/spec.md`，此刻已含 TTL 语义），把自己的 MODIFIED 重写成"在 TTL 语义之上叠加批量语义"的版本。这一步可以把冲突报告直接丢给 Cursor agent 处理：

```text
Cursor ▸ /hx-dev-design bulk-reserve
         rebase check 报告冲突如下，请基于当前主规格重写 MODIFIED 条目：
         CONFLICT stock-reservation/"预占创建" (MODIFIED): not found in base spec ...
```

`/hx-dev-design` 的 delta 定稿步骤要求在主规格漂移时执行 `hx rebase check`，并基于**当前**主规格重写 MODIFIED/REMOVED。agent 会读取含 TTL 语义的新版需求全文（含吴工新加的 Scenario），叠加批量语义后整体重写。郑工在 diff 里确认合并语义正确：

```markdown
## MODIFIED Requirements

### Requirement: 预占创建与过期
WHEN 收到单笔或批量（<=100 笔）预占请求, THE SYSTEM SHALL 为每笔创建带 TTL（默认 900 秒）的预占单，任一笔失败则整批回滚。

#### Scenario: reservation expires after ttl
（保留吴工的场景，原样带上）

#### Scenario: bulk reserve is atomic
- GIVEN 批量预占 3 笔，其中 1 笔库存不足
- WHEN 提交批量请求
- THEN 返回 409，3 笔均未创建
```

注意：**MODIFIED 必须携带该需求的完整最新内容**（包括别人刚加的 Scenario），因为合并是替换式的——这是"重写而非补丁"的代价，换来的是合并结果永远可读、无三方合并歧义。

改完重验：

```console
$ hx gate check bulk-reserve --stage dev --task design
GATE PASS (dev/design)
$ hx rebase check bulk-reserve
deltas apply cleanly against current specs
$ hx archive bulk-reserve
Archived to harnessX/archive/2026-07-08-bulk-reserve
Merged capabilities: stock-reservation, api-gateway
```

由于 delta 内容变了，CI 的 `meta verify` 会发现批准时的 artifact hash 与当前不符——所以郑工重写 delta 后需要**重新请张架构师批准 design-to-plan gate**。这不是官僚流程：规格实质变了，批准当然要重来。

## 关键机制解析

- **三道防线的层次**：创建时的域重叠**告警**（促成人际对齐）→ archive 前的 rebase **检查**（机器兜底）→ 冲突时**阻断并给出指引**（fail-closed）。轻的靠沟通，重的靠机器。
- **为什么不做自动三方合并？** 规格是给人和 agent 读的裁决依据，自动合并产生的"缝合文本"没人真正 review 过。强制"基于最新主规格重写"，保证每一版主规格都有明确的责任人。
- **批准哈希与冲突解决联动**：重写 delta ⇒ artifact hash 变化 ⇒ 原批准失效。机制自动保证"规格实质变更必然回到人工批准点"。
