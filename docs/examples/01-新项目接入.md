# 场景 01：新的后端 API 项目接入 HarnessX

## 背景

某电商公司新立项"优惠券服务"（coupon-service），Node.js + TypeScript 的分层架构（routes → services → repositories）。团队 5 人，其中 3 人重度使用 Cursor，1 人用 Claude Code，1 人用 Trae。技术负责人王工希望：**从第一天起，所有 AI 生成的代码都在同一套约束下交付**，而不是等混乱出现后再治理。

## 操作步骤

### 1. 初始化（选择 api-service 拓扑 Bundle）

```console
$ cd coupon-service
$ hx init --bundle api-service
Initialized /work/coupon-service/harnessX
  + constitution.md
  + config.yaml
  + harness.yaml
  + assets/
  + specs/
  + changes/
  + archive/
  + runs/
  + assets/bundles/api-service/

Next steps:
  1. Edit harnessX/constitution.md — write your project principles and core domains
  2. Create your first change:  hx change create <name> --domains <d1,d2>
  3. Draft the proposal:        hx propose <name> --title "..."
  4. Validate as you go:        hx gate check <name>
  5. Install enforcement:       hx hooks install && hx ci init
```

`--bundle api-service` 带来了三样东西（可用 `hx bundle list` 查看可选 Bundle）：

- `arch-boundary` 传感器 + `layering.yaml` 约束资产（成对出现：constraint 是前馈，sensor 是反馈）；
- `perf-budget` 传感器 + 性能预算 Skill；
- verification 套件被扩展为 `[spec-validate, spec-trace, arch-boundary, perf-budget]`。

### 2. 定制 Constitution（项目宪法）

编辑 `harnessX/constitution.md`，把优惠券的核心域写进去——这直接影响后续的 profile 推荐（见场景 03）：

```markdown
## Core domains

core-domains: [coupon-issuing, coupon-redemption]
```

再补一条团队自己的不可违背原则：

```markdown
6. 优惠券金额计算必须使用整数分（cent），任何浮点运算都是缺陷。
```

写完后跑一次静态自检，确认新原则没有与 Bundle 自带的 Skill 冲突：

```console
$ hx harness lint
no conflicting guide directives found
```

### 3. 安装本地强制（L2）与 CI 重放（L3）

```console
$ hx hooks install
installed /work/coupon-service/.git/hooks/pre-commit
installed /work/coupon-service/.git/hooks/pre-push

$ hx ci init
wrote /work/coupon-service/.github/workflows/harness-verify.yml
wrote /work/coupon-service/.github/BRANCH_PROTECTION.md
```

按 `BRANCH_PROTECTION.md` 的指引在 GitHub 上配置：

- `main` 分支必须通过 `harness-verify` 状态检查（gate 重放 + `meta verify` + `trace check` + `fixture verify`）；
- `harnessX/specs/` 与 `harnessX/changes/**/specs/` 加入 CODEOWNERS，指向 `@coupon-team/spec-owners`——**规格改动永远有人看**。

### 4. 编译多工具适配层（单源 → 多目标）

```console
$ hx adapter targets
cursor	Tier 1	commands,skills,rules,hooks,mcp,permissions
trae	Tier 1	commands,rules,agents,mcp
qoder	Tier 1	commands,skills,rules,agents,mcp
claude	Tier 1	commands,skills,rules,hooks,agents,mcp,permissions
generic	Tier 2	rules

$ hx adapter sync
cursor (Tier 1): 13 file(s)
  + .cursor/commands/hx-propose.md
  + .cursor/skills/coding-conventions/SKILL.md
  + .cursor/skills/spec-writing/SKILL.md
  + .cursor/rules/harnessx.mdc
  + .cursor/hooks.json
  ...
trae (Tier 1): 2 file(s)
claude (Tier 1): 12 file(s)
generic (Tier 2): 1 file(s)
```

`.cursor/commands/hx-propose.md` 这类命令文件的正文不是一行 CLI 桥接，而是从单源资产 `harnessX/assets/commands/propose.md` 编译来的**完整阶段工作流提示词**（步骤、护栏、完成标准）。开发者在 Cursor 里敲 `/hx-propose`，agent 拿到的就是这份阶段作业指导书，按其中的步骤起草 proposal、写 EARS 增量规格、跑 `hx gate check` 自检。

生成的文件全部带 `GENERATED by harnessx adapter v1.0.0 ... (hash:...)` 头。**团队约定：这些文件不许手改**，要改就改 `harnessX/assets/` 里的单源资产（命令提示词在 `assets/commands/`，技能在 `assets/guides/`）再重新 `adapter sync`。CI 里可以加一步 `hx adapter drift` 兜底（见场景 09）。

### 5. 首次提交

```console
$ git add -A && git commit -m "chore: bootstrap HarnessX"
[hx] pre-commit: running gate check
[hx] gate hook-check ok
```

pre-commit hook 已经生效——当前没有处于 implementing 状态的 change，所以直接放行。

## 关键机制解析

- **为什么用 Bundle 而不是逐个配置？** Bundle 是"拓扑级"打包：一个后端 API 服务需要的约束、传感器、套件组合是可复用的。新项目 30 秒获得与老项目一致的架构守护。
- **Constitution vs Skill 的分工**：宪法只放 5–10 条不可违背原则（冲突时的最高裁决依据），其余细则放 Skill。`hx harness lint` 会在两者矛盾时报警并指出按优先级链（constitution > profile > bundle > asset）谁生效。
- **三层强制力**：编辑器规则（L1，靠 agent 自觉）→ git hooks（L2，可 `--no-verify` 绕过）→ CI 重放（L3，绕不过）。三层都装上，约束才是"硬"的。
