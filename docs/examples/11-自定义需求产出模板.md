# 场景 11：开工前自定义需求产出模板（proposal + delta spec）
| | |
| --- | --- |
| **旅程** | 定制与迁移 |
| **适用角色** | 需求/BA |
| **前置** | 场景 01 |
| **关联** | 见 [场景选择指南](00-场景选择指南.md) |

## 背景

某金融科技公司的「账户服务」（account-service）已接入 HarnessX。合规要求每次行为变更的提案必须包含 **合规影响** 与 **数据分类** 两节；产品团队还希望 delta spec 里统一使用 `REQ-<域>-<序号>` 形式的需求标题，方便与 Jira 追溯。

平台工程师 **陈工** 要在团队写第一条 change **之前**，把需求阶段的产出模板和规范配好，而不是等 agent 按默认骨架写完再人工改格式。

角色分工：

- **陈工**：维护 `harnessX/assets/` 控制资产；
- **赵产品**：后续用 Cursor `/hx-propose` 起草 change，只填业务内容，不再重复解释章节结构。

## 操作步骤

### 1. 复制并扩展 proposal 模板

默认模板在 `harnessX/assets/guides/proposal-template/template.md`。陈工保留 Gate 强制检查的三节（`## Why`、`## What Changes`、`## Impact`），并增加合规段落：

```markdown
# Proposal: {{title}}

<!-- 提交前删除本行及所有指引注释 -->

## Why

<!-- 1–3 句：问题或机会。关联工单/事件。 -->

## What Changes

<!-- 每条可观察的行为变化；须能映射到 delta spec 中的一条 Requirement -->

- ...

## Impact

- Affected capabilities:
- Affected code:
- Breaking change: no
- Data classification: <!-- public | internal | confidential | restricted -->
- Compliance impact: <!-- 如无则写「无」 -->

## Out of Scope

<!-- 本 change 明确不做的事 -->

## Open Questions

<!-- 有歧义时列出，等人工答复；无则删除本节 -->
```

在 `examples/` 下增加一份「金标准」示例，供 agent few-shot 参考：

```console
$ cp harnessX/assets/guides/proposal-template/examples/add-rate-limiting.md \
     harnessX/assets/guides/proposal-template/examples/fintech-account-change.md
# 编辑 fintech-account-change.md，补上 Data classification / Compliance impact 示例段落
```

### 2. 定制 spec-writing Skill（delta spec 命名与 EARS 规范）

编辑 `harnessX/assets/guides/spec-writing/SKILL.md`，在文首追加团队命名约定（**不改变** OpenSpec Delta 结构与 EARS 校验规则）：

```markdown
## Requirement naming (account-service)

- ADDED/MODIFIED 需求标题格式：`Requirement: REQ-<DOMAIN>-<NNN> <short title>`
  例：`Requirement: REQ-ACCT-001 冻结账户余额查询`
- `<DOMAIN>` 与 `hx change create --domains` 声明一致（如 `account-balance` → `ACCT` 映射写在 constitution 或本 Skill 表内）。
- Scenario 名仍用英文 kebab-case；测试标题按 `Scenario: <name>` 字面匹配，勿随意改名。
```

EARS 句式、Scenario 块、MODIFIED 全文替换等硬性规则**保持不动**——它们由 `spec-validate` 传感器执行，不能通过改 Skill 绕过。

### 3. 确认 harness.yaml 注册（通常 init 已写好）

```yaml
guides:
  - id: proposal-template
    kind: guide.template
    execution: computational
    phase: [propose]
    source: assets/guides/proposal-template/template.md
  - id: spec-writing
    kind: guide.skill
    execution: inferential
    phase: [propose, spec]
    source: assets/guides/spec-writing/SKILL.md
```

若团队从 Hub 拉取了 `@org/base-harness`，可在 `harness.yaml` 用 `overrides` 指向本地定制版（须写 `reason` 备查）：

```yaml
overrides:
  - id: proposal-template
    source: assets/guides/proposal-template/template.md
    reason: "金融域合规：增加数据分类与合规影响字段"
```

### 4. 静态自检 + 编译到 Cursor

```console
$ hx harness lint
no conflicting guide directives found

$ hx adapter sync
cursor (Tier 1): 13 file(s)
  + .cursor/skills/spec-writing/SKILL.md
  ...
```

`hx harness lint` 确认新 Skill 条目不与 constitution 或 Bundle 约束矛盾；`adapter sync` 把模板与 Skill 编译进 `.cursor/skills/` 与斜杠命令，**团队成员无需各自复制规则**。

### 5. 验证：`hx propose` 使用新模板

赵产品创建 change 并起草提案：

```console
$ hx change create freeze-balance-query --domains account-balance
Created change "freeze-balance-query" (profile: standard, domains: account-balance)

$ hx propose freeze-balance-query --title "支持查询冻结余额"
wrote harnessX/changes/freeze-balance-query/proposal.md
wrote harnessX/changes/freeze-balance-query/specs/account-balance/spec.md
```

生成的 `proposal.md` 已含 **Data classification**、**Compliance impact**、**Open Questions** 骨架；`{{title}}` 被替换为「支持查询冻结余额」。

赵产品在 Cursor 中：

```text
Cursor ▸ /hx-propose freeze-balance-query
         需求：用户可查询账户当前冻结金额。PRD 见 @docs/prd/freeze-balance.md
```

agent 按编译后的 `/hx-propose` 提示词：填满各节、按 `REQ-ACCT-*` 命名写 delta spec、跑 `hx gate check --phase spec` 直到通过。

### 6. Gate 行为：什么能改、什么不能改

**可以自定义：**

- proposal 中除三节以外的任意章节（合规、开放问题、风险表等）；
- 模板内 HTML 注释形式的填写指引；
- `examples/` 下的正反例；
- spec-writing Skill 中的命名、覆盖范围说明。

**不能去掉或改名（否则 Gate 阻断）：**

```text
$ hx gate check freeze-balance-query --phase design
BLOCKER  proposal.md missing section "## Why"
GATE BLOCKED (design)
```

Gate 硬编码检查 `## Why`、`## What Changes`、`## Impact` 必须存在，且不能残留 `{{title}}` 占位符。`## Out of Scope` 在默认模板里有，但**不**在 Gate 强制列表中——建议保留作范围约束。

delta spec 方面：`spec-validate` 仍要求 ADDED/MODIFIED 需求含 EARS + Scenario，与模板定制无关。

## 关键机制解析

- **前馈**：`guide.template`（proposal-template）在 `hx propose` 时**确定性**渲染进 `proposal.md`；`guide.skill`（spec-writing）在 propose/spec 阶段进入 Context Pack，约束 agent 怎么写 delta spec。
- **反馈**：章节完整性由 `proposalProblems()` 检查；delta 格式由 `spec-validate` 检查——模板定制不能替代 Sensor。
- **单源**：只改 `harnessX/assets/`，再 `hx adapter sync`；不要手改 `.cursor/` 下 GENERATED 文件（见场景 09）。
- **分层**：组织 Hub 模板 → 团队 `overrides` → 仓库本地 `assets/`，越靠近业务仓库优先级越高（设计文档 §11.2）。

## 与后续阶段的关系

custom proposal 只影响 **propose/design/spec** 阶段的可读性与合规留痕；**spec→plan 人工批准**、traceability、archive 合并规则不变。design 阶段 agent 仍会在 Context Pack 里读到已填好的 `proposal.md`——因此 proposal 模板里写清楚的合规字段，会自然成为概要设计的输入约束。
