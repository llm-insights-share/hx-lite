## 背景

安全组维护一个内部敏感信息扫描器（Python 写的 `secscan`，检查硬编码密钥、内网地址、身份证号正则等）。他们希望：

1. 把 secscan 作为 Sensor 挂进各仓库的 verification 套件，**输出结构化报告**而不是一堆日志；
2. 对 `config/` 与 `*.env.example` 这类高危路径做**保存即扫**（不等到 verify）；
3. 扫出问题时，能一键生成带上下文的修复会话。

## 操作步骤

### 1. 用命令协议接入 Python 扫描器（不写一行 Node 代码）

插件命令协议：stdin 收 JSON 上下文，stdout 出 JSON 报告。给 secscan 包一层薄适配：

```python
#!/usr/bin/env python3
# harnessX/plugins/secscan_adapter.py
import json, sys, subprocess

ctx = json.load(sys.stdin)                      # {root, base, change, sensor:{id,kind,execution}}
raw = subprocess.run(["secscan", "--json", ctx["root"]], capture_output=True, text=True)
issues = json.loads(raw.stdout or "[]")

print(json.dumps({
    "status": "fail" if issues else "pass",
    "summary": f"{len(issues)} secret-like finding(s)",
    "findings": [{
        "severity": "block",
        "file": i["path"], "line": i["line"],
        "rule": i["rule"],
        "message": i["message"],
        "fix_hint": "移除硬编码值，改用环境变量或密钥管理服务；确属误报请加 waiver"
    } for i in issues],
    "fix_command": f"hx fix --change {ctx.get('change') or '<change>'} --sensor secscan"
}))
```

在 `harness.yaml` 注册（`plugin: cmd:` 前缀即命令协议；Node 插件则写模块路径，宿主会校验其 `api` 主版本兼容）：

```yaml
sensors:
  - id: secscan
    kind: sensor.script
    execution: computational
    phase: [verify]
    plugin: "cmd:python3 harnessX/plugins/secscan_adapter.py"
    on_fail: block
    timeout_ms: 180000

suites:
  verification: [spec-validate, spec-trace, arch-boundary, secscan]
```

**fail-closed 语义自动继承**：适配脚本崩溃、超时、输出的不是合法 JSON——都算 error，gate 一律阻断，绝不会"扫描器挂了 = 没有问题"。

### 2. 高危路径保存即扫（file-save 触发）

再注册一个事件驱动实例，scope 用 glob 圈定高危路径：

```yaml
  - id: secscan-hot
    kind: sensor.script
    execution: computational
    trigger: file-save
    scope: ["config/**", "**/*.env.example", "deploy/**"]
    plugin: "cmd:python3 harnessX/plugins/secscan_adapter.py"
    on_fail: block
```

开发机上起守护（或由编辑器 hooks 触发等价检查）：

```console
$ hx watch
watching for file-save triggered sensors (ctrl-c to stop)
[fail] secscan-hot ← config/redis.yaml: 1 secret-like finding(s)
```

李工刚把一个测试用 Redis 密码写进 `config/redis.yaml`，保存两秒后就被点名——**反馈离错误越近，纠正成本越低**。同时每晚 CI cron 跑 `hx schedule run`，执行所有 `trigger: schedule` 的传感器（全量深扫版 secscan、janitor 等共用这个调度入口）。

### 3. 一键修复回环：hx fix

verify 阶段 secscan 拦下 3 处硬编码后，报告里的 `fix_command` 可直接执行：

```console
$ hx fix --change fee-recalc --sensor secscan
fix pack: harnessX/changes/fee-recalc/fix-pack.md (3 finding(s))
```

`fix-pack.md` 是为修复会话组装的聚焦 Context Pack：每条 finding（含 fix_hint）+ 出错文件的现场代码片段 + 本 change 的 delta spec + "不许为了过检而删检查"的纪律说明。喂给 agent 有两种方式——在 Cursor 对话框里直接引用：

```text
Cursor ▸ 按 @harnessX/changes/fee-recalc/fix-pack.md 修复全部 3 处 finding，
         修完跑 hx gate check fee-recalc 复验
```

或 headless 挂 runner：

```console
$ hx fix --change fee-recalc --sensor secscan --runner 'cursor-agent --prompt-file "$HX_FIX_PACK"'
```

相比把整个仓库上下文丢给 agent"你看看哪里有密钥"，fix pack 的定向上下文既省 token 又不易跑偏。

### 4. 误报治理：waiver 而不是关规则

`deploy/seed-data.sql` 里一串测试身份证号被误报。正确处置不是把 secscan 从套件里摘掉，而是定向豁免 + 到期重审：

```console
$ hx waiver add fee-recalc --target secscan \
    --reason "seed-data.sql 中为造数用的假身份证号，已与安全组确认（SEC-1142）" \
    --requested-by li.dev --approved-by security.zhao \
    --expires 2026-10-01T00:00:00Z
waiver 8c2d91af added for secscan, expires 2026-10-01T00:00:00.000Z
```

waiver 生效期间该传感器的失败降级为警告（gate 输出 `warning secscan: ... (waived)`），过期后自动恢复阻断，janitor 每晚催办。同时安全组把这类误报模式记入 secscan 自身的白名单——**waiver 数据反过来驱动扫描器进化**。

### 5. 插件的版本纪律

安全组季度升级 secscan 输出格式时，把适配层当正式软件对待：插件协议随宿主 `PLUGIN_API_VERSION`（SemVer）演进，宿主承诺同一主版本内向后兼容；Node 插件声明 `api: "1.x"`，主版本不匹配会在加载时被拒绝，而不是运行到一半用错字段静默出错。

## 关键机制解析

- **两种接入姿势**：命令协议（任意语言，stdin/stdout JSON）适合已有工具的快速包装；Node 插件（`{api, id, execute}` 模块）适合需要深度访问工作区结构的场景。两者产出同一种 SensorReport，对 gate 完全透明。
- **触发时机是三档而非一档**：phase（gate 时）、file-save（保存即扫，scope 限定省资源）、schedule（夜间全量）。同一扫描能力按成本和时效分层部署。
- **结构化报告是修复回环的前提**：`findings[].file/line/fix_hint` + `fix_command` 让"发现问题→组装上下文→定向修复→复验"全链路无人工转录。日志式输出做不到这一点。
