## 背景

零售企业 **RetailCo** 的 **会员积分中心** 要在同一 monorepo 交付：

| 子系统 | 目录 | 拓扑 Bundle |
| --- | --- | --- |
| 积分 API（后端） | `apps/api/` | `api-service` |
| 运营后台（B 端） | `apps/admin-web/` | `frontend-dashboard` |
| 会员门户（C 端） | `apps/portal-web/` | `frontend-2c` |

业务目标：会员在门户查看积分余额与明细；运营在后台配置「消费 1 元得 N 分」规则；后端提供 REST API 并保证分层架构。

**角色配置（5 人）**

| 角色 | 人员 | 主要职责 | 常用入口 |
| --- | --- | --- | --- |
| 产品经理 | **陈产品** | 需求边界、proposal、delta spec、批准参与 | Cursor `/hx-propose` `/hx-spec`；终端 `hx gate approve` |
| 设计人员 | **林设计** | 交互与信息架构、API 清单、ADR、可观测性 | Cursor `/hx-design` |
| 前端开发 | **赵前端** | `apps/admin-web` + `apps/portal-web` 实现与 UI 测试 | Cursor `/hx-apply`（前端任务） |
| 后端开发 | **李后端** | `apps/api` 实现与 API 集成测试 | Cursor `/hx-apply`（后端任务） |
| 测试人员 | **周测试** | 测试用例评审、traceability、verify 收口 | 终端 `hx testfirst approve`；`/hx-verify` |

**Harness profile**：`standard`（含 design / 人工批准 / verify）。

---

## 执行顺序总览（按时间线）

| 顺序 | 阶段 | 主导角色 | 协作角色 | 产出物 | 门禁 |
| --- | --- | --- | --- | --- | --- |
| 0 | 仓库接入（一次性） | 李后端 + 赵前端 | 全员 | `harnessX/`、双前端 + API bundle | — |
| 1 | Propose | 陈产品 | 林设计（初评） | `proposal.md`、delta spec 草稿 | propose gate |
| 2 | Design | 林设计 | 陈产品、李后端 | `design.md`（API/数据/可观测性） | design gate |
| 3 | Spec | 陈产品 | 李后端、周测试 | 完整 delta spec（EARS + Scenario） | spec gate |
| 4 | 人工批准 | 陈产品 | 周测试 | `meta.yaml` 批准记录 | spec→plan 门 |
| 5 | Plan | 李后端 | 赵前端、周测试 | `tasks.md` 双轨 + 前后端并行组 | plan gate |
| 6 | Apply | 赵前端 + 李后端 | 周测试（评审测试） | 代码 + 测试 + traceability | apply fast 套件 |
| 7 | Verify | 周测试 | 全员修缺陷 | 全量验证通过 | verify gate |
| 8 | Archive | 陈产品 | 李后端 | 主规格合并、change 归档 | archive |

> **分工原则**：agent 能写的（提案、设计稿、规格、代码、修 lint）走 **Cursor 对话框**；只有人能做的（批准、签核、发布判断）走 **终端 `hx`**，输出即审计痕迹。

## 0. 仓库接入（一次性，全员知晓）

李后端在 monorepo 根目录初始化 Harness，**叠加多拓扑 bundle**：

```console
$ hx init --bundle api-service
$ hx bundle add frontend-dashboard
$ hx bundle add frontend-2c
$ hx adapter sync
$ hx hooks install && hx ci init
```

`harness.yaml` 将同时包含：

- `api-design` / `layering-rules`（后端）
- `ui-architecture`（后台 dashboard）
- `web-2c-architecture`（会员门户 SEO/LCP）

赵前端确认 `.cursor/commands/hx-*.md` 与 `apps/*/src` 目录约定；周测试确认 `tests/` 与 `Scenario:` 命名规范。

## 1. Propose — 陈产品（林设计列席）

```console
$ hx change create member-points --domains member,points
Created change "member-points" (profile: standard, domains: member, points)
```

```text
Cursor ▸ /hx-propose member-points
         PRD：会员积分中心 v1 — 门户查余额/明细；后台配置赚取规则；API 供两端调用。
         见 @docs/prd/member-points-v1.md
```

陈产品在 Cursor 中驱动 agent 完成：

1. `hx propose member-points --title "会员积分中心 v1"`
2. 填写 `proposal.md`（Why / What / Impact / Out of Scope）
3. 起草 `specs/member/spec.md`、`specs/points/spec.md` 的 ADDED 骨架

林设计列席：确认 Out of Scope 不含「积分兑换商城」（下期再做）。

```console
$ hx gate advance member-points    # proposed
GATE PASS (propose)
advanced: explore → proposed
```

## 2. Design — 林设计（陈产品、李后端评审）

```text
Cursor ▸ /hx-design member-points
         请按 design-template 填写：门户与后台的信息架构、积分 API 清单、
         规则表结构、指标与告警、回滚方案。
```

林设计产出 `design.md` 要点（agent 按模板扩写）：

- **API Surface**：`GET /v1/members/{id}/points`、`GET .../ledger`、`PUT /v1/admin/earn-rules`
- **Data Model**：`point_accounts`、`point_ledger`、`earn_rules`
- **ADR-1**：规则变更异步生效 vs 同步 — 选同步 + 版本号，便于两端一致
- **Architecture Constraints**：`apps/api` 遵守 routes→services→repositories；前端遵守各自 bundle 分层
- **Observability**：`points_balance_query_total`、`earn_rule_update_total`

李后端核对 API 与分层；陈产品确认 PRD 范围未膨胀。

```console
$ hx gate advance member-points    # → designed
$ hx gate advance member-points    # → specified（fast：spec-validate）
```

---

## 3. Spec — 陈产品（李后端、周测试共建）

```text
Cursor ▸ /hx-spec member-points
         补齐 member/points 两个 capability 的 delta spec；
         每个 Requirement 至少 2 个 Scenario（含错误路径）；
         周测试要求的边界：负数积分、规则不存在、未授权访问。
```

示例片段（`specs/points/spec.md`）：

```markdown
### Requirement: 查询积分余额
WHEN 已认证会员请求本人积分余额, THE SYSTEM SHALL 返回当前可用积分与冻结积分。

#### Scenario: member views own balance
- GIVEN 会员 A 可用积分 1200
- WHEN 会员 A 请求 GET /v1/members/A/points
- THEN 200，body.points.available 为 1200

#### Scenario: cross-member access denied
- GIVEN 会员 A 已登录
- WHEN 会员 A 请求 GET /v1/members/B/points
- THEN 403，错误码 MEMBER_FORBIDDEN
```

周测试在 PR 评论中标注：每个 Scenario 必须有对应测试名（字面包含 `Scenario:` 字符串）。

```console
$ hx gate check member-points --phase spec
GATE PASS (spec)
```

## 5. Plan — 李后端主导（赵前端、周测试复核 tasks）

```console

```

双轨任务示例（**前后端可并行组**）：

```markdown
- [ ] 01a [test] (points / Requirement: 查询积分余额) API tests @group=be-tests
- [ ] 02a [test] (points / Requirement: 配置赚取规则) API tests @group=be-tests
- [ ] 03a [test] (member / Requirement: 门户展示积分) Portal E2E @group=fe-portal-tests
- [ ] 04a [test] (member / Requirement: 后台配置规则) Admin UI tests @group=fe-admin-tests
- [ ] 01b [impl] (points / ...) Implement balance API @group=be-impl @depends=01a
- [ ] 02b [impl] (points / ...) Implement earn rules API @depends=02a
- [ ] 03b [impl] (member / ...) Portal points page @depends=03a
- [ ] 04b [impl] (member / ...) Admin earn rule form @depends=04a
```

周测试确认：**不得删除任何 `[test]` 任务**；每个 Scenario 在 tasks 中有引用。

```console

```

## 6. Apply — 赵前端 + 李后端（并行）

### 6a. 测试人员：评审并批准测试桩（可选 strict 路径）

周测试在新会话中审查 `[test]` 任务产出：

```console
$ hx testfirst approve member-points --files tests/api/points.test.ts,apps/portal-web/tests/points.spec.ts --by zhou.qa
```

### 6b. 后端 — 李后端

```text
Cursor ▸ /hx-apply member-points
         只执行 @group=be-tests 与 @group=be-impl 任务；代码仅限 apps/api/。
```

或终端并行（与前端同时进行）：

```console
$ hx apply member-points --parallel 2 --runner '<agent>' 
# 李后端负责盯后端 task；赵前端另开会话盯前端 task
```

每个任务后 **fast 套件**（spec-validate、typecheck、lint、unit-changed）自动运行；`apps/api` 受 `arch-boundary` 约束。

### 6c. 前端 — 赵前端

```text
Cursor ▸ /hx-apply member-points
         执行 fe-portal-tests / fe-admin-tests / 对应 impl；
         apps/admin-web 遵守 dashboard 分层；apps/portal-web 遵守 2C SEO 约束。
```

门户测试命名示例：

```typescript
it("Scenario: member views own balance", async () => { ... });
```

### 6d. 角色协作检查点

| 检查点 | 负责人 | 动作 |
| --- | --- | --- |
| API 契约与 design 一致 | 李后端 | `hx gate check member-points --phase apply` |
| UI 与 API 联调 | 赵前端 | 本地对接 staging API |
| 测试覆盖 | 周测试 | `hx trace check member-points` |


## 多角色协作要点

1. **单 change、双端代码**：前后端共用一个 `member-points` change，delta spec 按 capability 拆分；避免「前端无规格」。
2. **Bundle 各管各层**：API 用 `api-service`；后台/门户各用对应前端 bundle 的 `layering.yaml`，`arch-boundary` 在 verify 分 app 扫描。
3. **并行 apply**：`@group=` 让赵前端与李后端同时推进，但 **spec 与 meta.yaml 仍单源**。
4. **测试人员前置**：周测试在 spec 阶段介入 Scenario，在 plan 阶段守住 `[test]` 任务，在 verify 阶段用 traceability 一票否决。
5. **人工批准绑定 artifact hash**：陈产品批准后若有人改 spec，CI `meta verify` 会拦。
