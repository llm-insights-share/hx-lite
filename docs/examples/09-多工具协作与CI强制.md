# 场景 09：一个仓库、四种 AI 工具，以及绕不过去的 CI

## 背景

结算服务（settlement-service）团队工具偏好各异：李工用 Cursor、周工用 Claude Code、郑工用 Trae（喜欢它的规划/执行双 agent）、外包同学用 Qoder 跑长任务 Quest。此外还有人偶尔用 Copilot。

痛点：不能给每种工具各维护一套规则文件——那是四份注定漂移的拷贝。HarnessX 的答案：**`harnessX/assets/` 单源，adapter 编译到各工具原生格式**；生成物是"编译产物"，禁止手改。

## 操作步骤

### 1. 一次编译，各取所需

```console
$ hx adapter sync
cursor (Tier 1): 12 file(s)
trae (Tier 1): 2 file(s)
qoder (Tier 1): 4 file(s)
claude (Tier 1): 11 file(s)
generic (Tier 2): 1 file(s)
```

同一份命令集与规则，落到每个工具的原生位置。命令的**提示词正文**来自单源资产 `harnessX/assets/commands/<phase>.md`（`guide.command` 类型）：每个斜杠命令不是"提醒你去跑 CLI"的一行桥接，而是该阶段的完整工作流提示词——步骤、护栏（Guardrails）、完成标准，agent 在工具内执行 `/hx-propose` 时拿到的就是这份阶段作业指导书。

| 工具 | 生成物 | 说明 |
| --- | --- | --- |
| Cursor | `.cursor/commands/hx-*.md`、`.cursor/skills/*/SKILL.md`、`.cursor/rules/harnessx.mdc`、`.cursor/hooks.json` | 命令文件携带完整阶段工作流提示词；hooks 在提交提示前跑 `hx gate hook-check`，编辑 fixtures/meta.yaml 后跑 `hx fixture verify` |
| Claude Code | `CLAUDE.md`、`.claude/commands/hx-*.md`、`.claude/settings.json` | 命令文件同上；settings 里 **deny 掉对 meta.yaml / fixtures / lock 文件的编辑权限**（L2 强制） |
| Trae | `.trae/rules/project_rules.md`、`.trae/agents.yaml` | agents.yaml 定义 `hx-planner`（只许跑 propose/design/spec/plan）与 `hx-executor`（只许跑 apply/verify），命令白名单分离 |
| Qoder | `.qoder/rules/harnessx.md`、`.qoder/skills/*.md`、`.qoder/commands/hx-*.md`、`.qoder/mcp.json` | 另有 Quest 导出（见步骤 3） |
| 其他（Copilot 等） | `AGENTS.md` | Tier 2 兜底：规则 + 命令说明，至少保证"知道规矩" |

改命令工作流只改 `harnessX/assets/commands/` 里的单源文件，重跑 `hx adapter sync` 即同步到所有工具；同一份提示词也会进入对应阶段的 Context Pack（`hx guide pack`），CLI 侧与工具侧看到的是同一套作业指导。

### 2. 手改检测：编译产物不是编辑入口

周工图快，直接在 `CLAUDE.md` 里加了一条团队约定。CI 中的漂移检查立刻发现：

```console
$ hx adapter drift
DRIFT CLAUDE.md: manually-edited
```

每个生成文件头部有 `GENERATED by harnessx adapter v1.0.0 ... (hash:...)`，drift 检查重算正文哈希比对。正确姿势：把那条约定写进 `harnessX/assets/guides/` 下的 Skill（或宪法），然后 `hx adapter sync`——**一处修改，五处生效**。

### 3. Qoder 长任务：Quest 导出与 worktree 隔离

外包同学接手一条已 plan 完的 change（`fee-recalc`，7 个任务）。把 delta spec + tasks 打包成 Quest 输入：

```console
$ hx adapter quest fee-recalc
wrote .qoder/quests/fee-recalc.md
```

Quest 文件包含完整规格、任务清单（含 Requirement 标注）和环境指引（`git worktree add ../quest-fee-recalc` 隔离运行，完成后在 worktree 内 `hx verify fee-recalc`）。**worktree 隔离 + verify 收口**意味着：不管 Qoder 在里面怎么折腾，进主干的唯一通道仍是全绿的 verification 套件。

### 4. 绕过演习：`--no-verify` 走不远

郑工赶下班，本地 fast 套件红着，强行绕过 hooks 提交：

```console
$ git commit -m "wip: fee recalc" --no-verify
$ git push --no-verify
```

推是推上去了（L2 本来就可绕）。但 PR 上的 `harness-verify` 工作流（L3）重放了所有检查：

```console
Replay gates for active changes
  fee-recalc [apply]: BLOCKED
BLOCKER  lint: exit 1
GATE BLOCKED (apply)
Error: Process completed with exit code 1.
```

分支保护要求该状态检查必过，PR 合不了。**本地 hooks 是给开发者省时间的（早发现），CI 重放才是强制力的来源。**

### 5. 更隐蔽的绕过：直接改 meta.yaml

有 agent（或人）发现 gate 状态存在 `meta.yaml` 里，试图手改 `status: implementing` 为 `status: verified` 蒙混过关。CI 的下一步就是干这个的：

```console
Verify meta.yaml integrity (FR-050)
  fee-recalc: TAMPERED
  - meta.yaml contentHash mismatch — file was edited outside the hx CLI
Error: Process completed with exit code 1.
```

meta.yaml 只有 hx CLI 会写，写入时盖 contentHash；gate 记录还绑定传感器日志的前缀哈希——改状态、改历史日志，两条路都被堵死。配合 Claude 的 permissions deny 与 Cursor hooks（L2 提前拦），这类"改账本"行为在三层上都有防线。

## 关键机制解析

- **能力声明而非厂商硬编码**：`hx adapter targets` 显示每个工具声明的能力（commands/skills/rules/hooks/agents/mcp/permissions），Tier 由能力算出。新工具接入只需写一个 emitter + 能力声明，不动核心。
- **"单源 + 编译 + 漂移检测"三件套**：等价于"不要手改 build 产物"的工程常识在提示词资产上的应用。没有漂移检测，单源承诺三周内必然瓦解。
- **异构工具下的统一收口**：不同工具的 agent 能力参差（有的支持 hooks 有的不支持），但所有路径最终都汇聚到同一组 CLI gate——`hx verify` 全绿才算数。工具是可换的，收口是唯一的。
