# 场景 02：订单服务新增"部分退款"功能（standard profile 全流程）
| | |
| --- | --- |
| **旅程** | 入门·日常 |
| **适用角色** | 后端开发 |
| **前置** | 场景 01 |
| **关联** | 见 [场景选择指南](00-场景选择指南.md) |

## 背景

订单服务（order-service）已接入 HarnessX，主规格 `harnessX/specs/order-refund/spec.md` 中已有"全额退款"的既有需求。产品要求支持**部分退款**：一笔订单可多次退款，累计退款额不得超过实付金额。

角色分工：

- **李工**（后端开发）：负责该 change，日常用 Cursor 的 agent 写代码；
- **张架构师**：design→plan 阶段的人工批准人（CODEOWNERS 中 `harnessX/changes/**/specs/` 的 owner）。

## 操作步骤

### 1. 创建 change 并起草提案（Cursor 对话框驱动）

李工在终端建好 change 工作区：

```console
$ hx change create partial-refund --domains order-refund
Created change "partial-refund" (profile: standard, domains: order-refund)
```

然后切到 Cursor 的 Agent 对话框，用斜杠命令驱动 propose 阶段：

```text
Cursor ▸ /hx-dev-propose partial-refund
         需求：支持部分退款。一笔订单可多次退款，累计退款额不得超过实付金额。
         产品文档见 @docs/prd/partial-refund.md
```

`/hx-dev-propose` 的正文（编译自 `harnessX/assets/commands/propose.md`）会指挥 agent 依次完成：

1. 执行 `hx propose partial-refund --title "支持部分退款"` 生成脚手架（proposal.md + delta spec 骨架）；
2. 结合李工给的 PRD 填满 proposal.md 的 Why / What Changes / Impact / Out of Scope 四节，删掉模板占位符——**不填完整就过不了后面的 design gate**；
3. 先读主规格 `harnessX/specs/order-refund/spec.md` 里"全额退款"的现有文本，再写 delta spec（spec-writing Skill 此时被自动挂载，约束 EARS 句式与场景命名）；
4. 跑 `hx gate check partial-refund --stage dev --task propose` 自检，不绿不收工。

薄命令 Guardrails（加 adapter 附录）同时约束 agent：本阶段不写实现代码；不发明 PRD 里没有的需求（放 Out of Scope）；有歧义列在 proposal.md 顶部问人，而不是替李工做决定。

### 2. 编写 delta spec（EARS 句式）

agent 产出的 `harnessX/changes/partial-refund/specs/order-refund/spec.md`（李工在 Cursor 的 diff 视图里逐段确认）：

```markdown
# Delta for order-refund

## ADDED Requirements

### Requirement: 部分退款
WHEN 收到金额为 X 的退款请求且 X + 已退款总额 <= 订单实付金额, THE SYSTEM SHALL 创建一笔金额为 X 的退款单并返回 201。

#### Scenario: partial refund accepted
- GIVEN 实付 100 元、已退 30 元的订单
- WHEN 请求退款 50 元
- THEN 创建退款单，订单累计退款 80 元

#### Scenario: refund exceeding paid amount rejected
- GIVEN 实付 100 元、已退 30 元的订单
- WHEN 请求退款 80 元
- THEN 返回 422，错误码 REFUND_EXCEEDS_PAID

## MODIFIED Requirements

### Requirement: 全额退款
WHEN 收到全额退款请求, THE SYSTEM SHALL 将其视为金额等于剩余可退金额的部分退款处理。

#### Scenario: full refund reuses partial path
- GIVEN 实付 100 元、已退 30 元的订单
- WHEN 请求全额退款
- THEN 创建 70 元退款单
```

写坏格式会被立刻拦住。比如 agent 第一版忘了给"部分退款"写第二个 Scenario、响应里写了"尽快退款"这种不可测度量，它按附录中的 Sensor/Gate 提醒自检时就会撞上：

```console
$ hx gate check partial-refund --stage dev --task propose
BLOCKER  spec-validate: 1 spec problem(s) across 1 delta file(s)
GATE BLOCKED (dev/propose)
```

传感器报告（写入 `runs/telemetry.jsonl`，也是 `hx fix` 的原料）中每条 finding 都带 `fix_hint`，agent 按提示改完再跑即通过——这个"写→检→修"循环发生在同一个 Cursor 会话内，不需要李工介入。

### 3. dev 阶段推进：propose → design → design-to-plan 批准 → plan

design 任务同样在 Cursor 里驱动：

```text
Cursor ▸ /hx-dev-design partial-refund
```

agent 按 `/hx-dev-design` 薄清单先跑 `hx design partial-refund`（该命令内部先做 propose 完整性 gate 检查），然后填写 `design/overview.md`（及 LLD 分包）：Context、ADR（本例的关键决策是"全额退款复用部分退款路径"，含被否掉的备选方案及原因）、可被传感器机械检查的架构约束。李工 review 后在终端推进任务（`meta.yaml` 记录 `stage`/`task`，gate 推进属于管控面动作，留在终端做，输出即审计痕迹）：

```console
$ hx gate advance partial-refund        # → dev/design（前置：proposal 完整）
GATE PASS (dev/design)
advanced: dev/propose → dev/design
```

完成 design 后，想直接进 plan 会被人工批准门拦住：

```console
$ hx gate advance partial-refund
BLOCKER  design→plan requires human approval: hx gate approve <change> --gate design-to-plan --approver <name> (FR-012)
GATE BLOCKED (dev/plan)
```

张架构师 review 了 delta spec（GitHub 上 CODEOWNERS 也会强制这次 review），确认"全额退款复用部分退款路径"的设计没问题后批准：

```console
$ hx gate approve partial-refund --gate design-to-plan --approver zhang.arch
approved gate "design-to-plan" by zhang.arch at 2026-07-04T09:12:33.201Z (artifact 3f6b1c9a02d4)
```

注意输出里的 **artifact hash**：批准绑定的是"批准那一刻的规格内容"。如果之后有人偷偷改 delta spec，CI 的 `meta verify` 阶段对不上账。

```console
$ hx plan partial-refund
wrote harnessX/changes/partial-refund/tasks.md (6 tasks)

$ hx gate advance partial-refund
GATE PASS (dev/plan)
advanced: dev/design → dev/plan
```

tasks.md 是双轨的——每个 Requirement 生成一个测试任务(a)和一个实现任务(b)：

```markdown
- [ ] 01a [test] (order-refund / Requirement: 部分退款) Write failing test(s) for scenarios of "部分退款"
- [ ] 01b [impl] (order-refund / Requirement: 部分退款) Implement "部分退款" until its tests pass
- [ ] 02a [test] (order-refund / Requirement: 全额退款) ...
```

生成后的任务清单可以让 agent 复核（`/hx-dev-plan partial-refund`（薄清单 + change-planning 附录）要求复核：地基任务排前面、过大任务拆分、`design/overview.md` 的每条 ADR 后果都要有对应任务、**不许删任何 [test] 任务**）。

### 4. apply：逐任务驱动 + 快速套件自校正

apply 有两种驱动方式，产出和约束完全一致。

**方式 A：Cursor 对话框内交互驱动**（适合边做边看）：

```text
Cursor ▸ /hx-dev-apply partial-refund
```

`/hx-dev-apply`（薄清单 + coding/apply 附录）要求 agent 严格按 tasks.md 顺序逐任务工作：每个 `[test]` 任务先写红测试（测试名含 `Scenario:` 原文）；每个 `[impl]` 任务只实现到让该场景测试变绿；**每完成一个任务跑一次 `hx gate check partial-refund --stage dev --task apply`（fast 套件），失败读 `fix_hint` 修复重跑，绝不通过弱化测试或删断言过检**；通过后在 tasks.md 勾掉该任务再进入下一个。李工在会话里能看到每个任务的 diff，随时打断纠偏。

**方式 B：终端 headless 循环**（适合挂机批量跑）。把 Cursor 的 CLI（`cursor-agent`）挂进 apply 循环，`$HX_TASK_*` 环境变量携带任务上下文，`$HX_FIX_HINTS` 携带上一轮失败提示：

```console
$ hx apply partial-refund --runner 'cursor-agent --prompt-file <(hx guide pack partial-refund --stage dev --task apply --out /dev/stdout) --task "$HX_TASK_TITLE"'
（runner 输出直通终端；01b 首次 lint 未过，第二轮通过 $HX_FIX_HINTS 携带失败提示自校正后通过）
completed tasks: 01a, 01b, 02a, 02b, 03a, 03b; remaining: 0
```

两种方式下，fast 套件（lint/typecheck/spec-validate 等，按 `harness.yaml` 配置）都在每个任务后自动运行；失败时 agent 拿着 `fix_hint` 自校正，最多 N 次（`--max-retries`，默认 3），超限则停下等人。期间 Cursor 的 `.cursor/hooks.json` 也在暗中值守：agent 若碰了 `tests/fixtures/` 下的文件，`afterFileEdit` 触发的 `hx fixture verify` 会当场报红。

**v0.2 可选：并行任务** — 若多个 `[test]` 任务彼此独立，可在 `tasks.md` 标注 `@group=` 并提高并发度（详见[场景 13](13-v0.2-编排与并行交付.md)）：

```markdown
- [ ] 01a [test] (...) Write failing test(s)... @group=tests-a
- [ ] 02a [test] (...) Write failing test(s)... @group=tests-a
- [ ] 01b [impl] (...) Implement... @depends=01a
```

```console
$ hx apply partial-refund --parallel 2 --runner '<agent>'
```

测试文件里用 `Scenario:` 命名建立可追溯性：

```typescript
it("Scenario: refund exceeding paid amount rejected", async () => { ... });
```

### 5. verify → archive

```console
$ hx verify partial-refund
VERIFIED

$ hx archive partial-refund
Archived to harnessX/archive/2026-07-04-partial-refund
Merged capabilities: order-refund
```

如果某个 Scenario 忘了写测试，verify 会精确指出：

```console
BLOCKER  uncovered scenario "full refund reuses partial path" (order-refund/全额退款) — add a test containing "Scenario: full refund reuses partial path" or a waiver
NOT VERIFIED
```

这时回到 Cursor 让 agent 收尾即可——`/hx-dev-verify partial-refund`（附录 / release-readiness）引导区分两种情况（测试存在但没引用场景串 → 补引用；测试真缺 → 补测试），修完自己重跑 `hx verify`。archive 是把 delta 合入主规格的管控面动作，由李工在终端执行。

archive 做了三件事：MODIFIED 的"全额退款"覆盖了主规格里的旧文本、ADDED 的"部分退款"追加进去；change 目录整体移入 `archive/2026-07-04-partial-refund/`；生成 `retro.md`（本次交付中传感器失败分布，作为 Steering 输入，见场景 07）。

## 关键机制解析

- **规格先行的意义**：agent 写代码前拿到的 Context Pack（`hx guide pack --stage dev --task apply`）只包含本 change 的 design/tasks/delta spec + 宪法 + apply 阶段的 Skill，**不含其他 change 的噪音**——这是控制上下文质量的前馈手段。
- **人工批准批的是"意图"**：机器验证实现（测试、lint、架构），人只在 design→plan 这一个瓶颈点确认"我们要做的是这件事"。批准带 artifact hash，事后改规格会被发现。
- **自校正上限**：无上限的重试会让 agent 在错误方向上越走越远。超限停下 + `hx fix --change partial-refund --sensor lint` 生成聚焦修复包，是更经济的恢复路径。
