# 场景 02：订单服务新增"部分退款"功能（standard profile 全流程）

## 背景

订单服务（order-service）已接入 HarnessX，主规格 `harnessX/specs/order-refund/spec.md` 中已有"全额退款"的既有需求。产品要求支持**部分退款**：一笔订单可多次退款，累计退款额不得超过实付金额。

角色分工：

- **李工**（后端开发）：负责该 change，日常用 Cursor 的 agent 写代码；
- **张架构师**：spec→plan 阶段的人工批准人（CODEOWNERS 中 `harnessX/changes/**/specs/` 的 owner）。

## 操作步骤

### 1. 创建 change 并起草提案

```console
$ hx change create partial-refund --domains order-refund
Created change "partial-refund" (profile: standard, domains: order-refund)

$ hx propose partial-refund --title "支持部分退款"
Wrote harnessX/changes/partial-refund/proposal.md
Wrote harnessX/changes/partial-refund/specs/order-refund/spec.md
```

李工让 Cursor agent 基于 `/hx-propose` 命令补全提案。生成的 proposal.md 模板要求填 Why / What Changes / Impact / Out of Scope 四节——**agent 不填完整就过不了后面的 design gate**。

### 2. 编写 delta spec（EARS 句式）

李工编辑 `harnessX/changes/partial-refund/specs/order-refund/spec.md`：

```markdown
# Delta for order-refund

## ADDED Requirements

### Requirement: 部分退款
WHEN 收到金额为 X 的退款请求且 X + 已退款总额 <= 订单实付金额, THE SYSTEM SHALL 创建一笔金额为 X 的退款单并返回 201。

#### Scenario: partial refund accepted
- GIVEN 实付 100 元、已退 30 元的订单
- WHEN 请求退款 50 元
- THEN 创建退款单，订单累计退款 80 元

#### Scenario: refund exceeding paid amount rejected
- GIVEN 实付 100 元、已退 30 元的订单
- WHEN 请求退款 80 元
- THEN 返回 422，错误码 REFUND_EXCEEDS_PAID

## MODIFIED Requirements

### Requirement: 全额退款
WHEN 收到全额退款请求, THE SYSTEM SHALL 将其视为金额等于剩余可退金额的部分退款处理。

#### Scenario: full refund reuses partial path
- GIVEN 实付 100 元、已退 30 元的订单
- WHEN 请求全额退款
- THEN 创建 70 元退款单
```

写坏格式会被立刻拦住。比如李工第一版忘了给"部分退款"写第二个 Scenario、响应里写了"尽快退款"这种不可测度量：

```console
$ hx gate check partial-refund --phase spec
BLOCKER  spec-validate: 1 spec problem(s) across 1 delta file(s)
GATE BLOCKED (spec)
```

传感器报告（写入 `runs/telemetry.jsonl`，也是 `hx fix` 的原料）中每条 finding 都带 `fix_hint`，agent 按提示改完再跑即通过。

### 3. 阶段推进：design → spec → 人工批准 → plan

```console
$ hx gate advance partial-refund        # → designed（前置：proposal 完整）
GATE PASS (design)
advanced: proposed → designed

$ hx design partial-refund              # 生成 design.md（ADR + 架构约束）
wrote harnessX/changes/partial-refund/design.md

$ hx gate advance partial-refund        # → specified（跑 fast 套件：spec-validate）
GATE PASS (spec)
advanced: designed → specified
```

此时想直接进 plan 会被人工批准门拦住：

```console
$ hx gate advance partial-refund
BLOCKER  spec→plan requires human approval: hx gate approve <change> --gate spec --approver <name> (FR-012)
GATE BLOCKED (plan)
```

张架构师 review 了 delta spec（GitHub 上 CODEOWNERS 也会强制这次 review），确认"全额退款复用部分退款路径"的设计没问题后批准：

```console
$ hx gate approve partial-refund --gate spec --approver zhang.arch
approved gate "spec" by zhang.arch at 2026-07-04T09:12:33.201Z (artifact 3f6b1c9a02d4)
```

注意输出里的 **artifact hash**：批准绑定的是"批准那一刻的规格内容"。如果之后有人偷偷改 delta spec，CI 的 `meta verify` 阶段对不上账。

```console
$ hx plan partial-refund
wrote harnessX/changes/partial-refund/tasks.md (6 tasks)

$ hx gate advance partial-refund
GATE PASS (plan)
advanced: specified → planned
```

tasks.md 是双轨的——每个 Requirement 生成一个测试任务(a)和一个实现任务(b)：

```markdown
- [ ] 01a [test] (order-refund / Requirement: 部分退款) Write failing test(s) for scenarios of "部分退款"
- [ ] 01b [impl] (order-refund / Requirement: 部分退款) Implement "部分退款" until its tests pass
- [ ] 02a [test] (order-refund / Requirement: 全额退款) ...
```

### 4. apply：逐任务驱动 + 快速套件自校正

李工把 Cursor agent 的 CLI 挂进 apply 循环（`$HX_TASK_*` 环境变量携带任务上下文，`$HX_FIX_HINTS` 携带上一轮失败提示）：

```console
$ hx apply partial-refund --runner 'cursor-agent --prompt-file <(hx guide pack partial-refund --phase apply --out /dev/stdout) --task "$HX_TASK_TITLE"'
（runner 输出直通终端；01b 首次 lint 未过，第二轮通过 $HX_FIX_HINTS 携带失败提示自校正后通过）
completed tasks: 01a, 01b, 02a, 02b, 03a, 03b; remaining: 0
```

每完成一个任务，fast 套件（lint/typecheck/spec-validate 等，按 `harness.yaml` 配置）自动运行；失败时 agent 拿着 `fix_hint` 自校正，最多 N 次（`--max-retries`，默认 3），超限则停下等人。

测试文件里用 `Scenario:` 命名建立可追溯性：

```typescript
it("Scenario: refund exceeding paid amount rejected", async () => { ... });
```

### 5. verify → archive

```console
$ hx verify partial-refund
VERIFIED

$ hx archive partial-refund
Archived to harnessX/archive/2026-07-04-partial-refund
Merged capabilities: order-refund
```

如果某个 Scenario 忘了写测试，verify 会精确指出：

```console
BLOCKER  uncovered scenario "full refund reuses partial path" (order-refund/全额退款) — add a test containing "Scenario: full refund reuses partial path" or a waiver
NOT VERIFIED
```

archive 做了三件事：MODIFIED 的"全额退款"覆盖了主规格里的旧文本、ADDED 的"部分退款"追加进去；change 目录整体移入 `archive/2026-07-04-partial-refund/`；生成 `retro.md`（本次交付中传感器失败分布，作为 Steering 输入，见场景 07）。

## 关键机制解析

- **规格先行的意义**：agent 写代码前拿到的 Context Pack（`hx guide pack --phase apply`）只包含本 change 的 design/tasks/delta spec + 宪法 + apply 阶段的 Skill，**不含其他 change 的噪音**——这是控制上下文质量的前馈手段。
- **人工批准批的是"意图"**：机器验证实现（测试、lint、架构），人只在 spec→plan 这一个瓶颈点确认"我们要做的是这件事"。批准带 artifact hash，事后改规格会被发现。
- **自校正上限**：无上限的重试会让 agent 在错误方向上越走越远。超限停下 + `hx fix --change partial-refund --sensor lint` 生成聚焦修复包，是更经济的恢复路径。
