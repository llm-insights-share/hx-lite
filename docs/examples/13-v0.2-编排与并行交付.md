# 场景 13：v0.2 编排能力 — 并行 Apply、Best-of-N、Diff 标注与 Issue 接入

## 背景

库存服务（inventory-service）团队升级到 **v0.2** 后，在 HarnessX 管控面之上增加了**执行编排**能力：同一 change 可并行跑多个 agent 任务、在隔离 worktree 里试跑多份实现、把 Code Review 的行级意见直接喂回 apply 循环。

本场景覆盖 v0.2 新增的编排能力：

| 能力 | CLI | 用途 |
| --- | --- | --- |
| GitHub Issue 脚手架 | `hx change create --from-issue` | 从 Issue 一键开 change |
| 并行任务 Apply | `hx apply --parallel` + `@group=` | 同组独立测试任务并发写 |
| Best-of-N fan-out | `hx apply --fan-out` | 多 worktree 试跑，选 verify 最优 |
| Worktree 运行时 | `hx runtime worktree` | 手动创建/管理隔离环境 |
| Diff 标注反馈 | `hx review import` | 行级 review 意见 → `HX_FIX_HINTS` |
| Guide 行为评测 | `hx eval guides` | 上线前验证 Context Pack 质量 |
| 变更通知 | `hx notify` | 长任务完成 / 待批准提醒 |

角色：

- **陈工**（后端）：主开发，在终端 / Cursor 里挂 CLI agent 跑 apply；
- **刘 PM**：在 GitHub 提 Issue，触发 change 创建；
- **赵 Reviewer**：Code Review 后导出标注 JSON，导入 HarnessX。

## 操作步骤

### 1. 从 GitHub Issue 创建 change（v0.2）

刘 PM 提了 Issue #87「库存预占超时自动释放」：

```console
$ hx change create reserve-timeout --from-issue https://github.com/acme/inventory-service/issues/87
Created change "reserve-timeout" from issue #87
  proposal: harnessX/changes/reserve-timeout/proposal.md
  delta: harnessX/changes/reserve-timeout/specs/inventory/spec.md
```

HarnessX 调 GitHub API 拉取标题与正文，预填 `proposal.md` 的 **Source Issue** 节，并按 Issue labels 推断 `touchedDomains`（无 label 时默认 `core`）。陈工也可显式指定域：

```console
$ hx change create reserve-timeout --from-issue https://github.com/acme/inventory-service/issues/87 --domains inventory,reservation
```

之后在 Cursor 对话框或终端走常规 propose→spec 流程补全 delta spec（见场景 02）。推进到 planned 后进入编排阶段。

### 2. 在 tasks.md 标注并行组与依赖（v0.2）

`hx plan reserve-timeout` 生成双轨任务后，陈工给**彼此独立**的测试任务加 `@group=`，实现任务加 `@depends=`：

```markdown
# Tasks: reserve-timeout

- [ ] 01a [test] (inventory / Requirement: 预占超时释放) Write failing test(s)... @group=tests-a
- [ ] 02a [test] (inventory / Requirement: 预占续期) Write failing test(s)... @group=tests-a
- [ ] 01b [impl] (inventory / Requirement: 预占超时释放) Implement... @depends=01a
- [ ] 02b [impl] (inventory / Requirement: 预占续期) Implement... @depends=02a
```

语义：

- `@group=tests-a`：01a 与 02a **无相互依赖**，可在同一批并发执行；
- 无 `@group` 的任务始终**串行**（一次一个）；
- `@depends=01a`：01b 必须等 01a 完成（impl 永远在对应 test 之后）。

### 3. 并行 Apply（v0.2）

陈工在主仓库 worktree 里跑：

```console
$ hx apply reserve-timeout --parallel 2 --runner 'codex exec --prompt "$HX_TASK_TITLE"'
completed tasks: 01a, 02a, 01b, 02b; remaining: 0
```

调度器行为：

1. 第一批：ready 且同组 → `[01a, 02a]` 并发（最多 2）；
2. 第二批：`01b`（depends 01a 已 done）；
3. 第三批：`02b`（depends 02a 已 done）。

每个任务完成后仍跑 profile 的 **fast 套件**；失败时 `$HX_FIX_HINTS` 携带 sensor 的 `fix_hint` 进入自校正（与 v0.1 一致，默认 `--max-retries 3`）。

### 4. Best-of-N：agent 不稳定时用 fan-out（v0.2）

预占逻辑复杂，单次 apply 偶尔过不了 verify。陈工改用 fan-out，让 HarnessX 在 N 个隔离 worktree 里各跑一遍 apply+verify，自动记分选优：

```console
$ hx apply reserve-timeout --fan-out 3 --runner 'claude -p "Implement task: $HX_TASK_TITLE"'
fan-out: 3 worktrees for reserve-timeout
  fanout-1: score=1000 verify=pass blockers=0
  fanout-2: score=1000 verify=pass blockers=0
  fanout-3: score=98 verify=fail blockers=2
selected: fanout-1 (/path/to/../hx-reserve-timeout-fanout-1)
```

结果写入 `harnessX/changes/reserve-timeout/runtime.yaml`：

```yaml
fanOut:
  at: "2026-07-05T02:00:00.000Z"
  count: 3
  candidates:
    - slot: fanout-1
      path: /tmp/.../hx-reserve-timeout-fanout-1
      verifyPassed: true
      blockerCount: 0
      score: 1000
  selected: fanout-1
worktrees:
  - slot: fanout-1
    path: ...
    branch: cursor/hx-reserve-timeout-fanout-1-5afb
```

**进主干的规则不变**：陈工在选中的 worktree 里确认 diff，合并回主分支；或手动把 fanout-1 的改动 cherry-pick 回来。未选中的 worktree 可清理：

```console
$ hx runtime worktree list reserve-timeout
fanout-1    /path/to/../hx-reserve-timeout-fanout-1    cursor/hx-reserve-timeout-fanout-1-5afb
fanout-2    ...
fanout-3    ...

$ hx runtime worktree remove reserve-timeout --slot fanout-3
worktree removed
```

也可不用 fan-out，手动开 worktree 后各自跑 apply（与场景 09 的 Qoder Quest 隔离思路相同，v0.2 提供了统一 CLI）：

```console
$ hx runtime worktree create reserve-timeout --slot wt-a
worktree wt-a: /path/to/../hx-reserve-timeout-wt-a (branch cursor/hx-reserve-timeout-wt-a-5afb)
```

在 `wt-a` 目录内同样执行 `hx apply reserve-timeout --runner '...'`，完成后 `hx verify reserve-timeout`；**verify 全绿**仍是合并前提。

### 5. Diff 行级标注 → fix_hints（v0.2）

赵 Reviewer 在 GitHub PR 上 review 后，把行级意见整理为 `reviews/reserve-timeout.json`（HarnessX 原生 JSON 格式）：

```json
[
  {
    "file": "src/reservation/timeout.ts",
    "line": 48,
    "severity": "critical",
    "comment": "TTL 必须从配置读取，禁止硬编码 300"
  },
  {
    "file": "src/reservation/timeout.ts",
    "line": 72,
    "severity": "important",
    "comment": "释放后应 emit ReservationReleased 事件"
  }
]
```

导入 HarnessX：

```console
$ hx review import reserve-timeout reviews/reserve-timeout.json
imported 2 annotation(s)
  hint: [CRITICAL] src/reservation/timeout.ts:48: TTL 必须从配置读取，禁止硬编码 300
  hint: [IMPORTANT] src/reservation/timeout.ts:72: 释放后应 emit ReservationReleased 事件
```

陈工继续 apply 或让 agent 修代码时，runner 环境的 **`HX_FIX_HINTS`** 会带上这些标注（与 sensor fix_hint 合并）。修完后：

```console
$ hx review resolve reserve-timeout import-1
resolved import-1
```

### 6. Guide 行为评测 — 上线前检查提示词质量（v0.2）

平台组改动了 `assets/commands/apply.md`，合并前跑 eval 确保 Context Pack 没丢 guide、没混入错误 phase 的噪音：

```console
$ hx eval guides reserve-timeout
propose-has-template: PASS
apply-no-proposal-noise: PASS
spec-has-spec-writing: PASS
plan-phase-scoped: PASS
all guide evals passed
```

用例来自 `packages/core/evals/guide-behavior.json`；团队可扩展：

```console
$ hx eval guides reserve-timeout --cases harnessX/evals/team-cases.json
```

失败示例（故意删了 apply command guide 后）：

```console
apply-no-proposal-noise: FAIL
  missing guide: cmd-apply
```

这测的是 **HarnessX 阶段 Context Pack 组装**是否正确——与 Superpowers 的 skill-behavior eval 思路类似，但针对本仓库的 guide 注册表。

### 7. 长任务通知（v0.2）

外包同学在隔离 worktree 里跑长 apply，陈工想在外出时收到「任务做完 / 待批准」提醒：

```console
$ hx notify reserve-timeout --once
watching reserve-timeout (implementing, tasks 2/4)
[hx watch] reserve-timeout needs_approval: Spec gate needs human approval before plan
```

持续轮询 + Webhook（Slack/飞书 等）：

```console
$ export HX_WATCH_WEBHOOK=https://hooks.example.com/harnessx
$ hx notify reserve-timeout --interval 60000
```

事件类型：`needs_approval`、`tasks_complete`、`review_pending`、`status_change`。

> **命名**：`hx watch` 仍是 file-save 传感器守护进程（场景 10）；变更轮询用 **`hx notify`**。

### 8. 收口：verify → archive

无论经过并行 apply 还是 fan-out，**唯一进主干通道**仍是：

```console
$ hx verify reserve-timeout
VERIFIED

$ hx archive reserve-timeout
Archived to harnessX/archive/2026-07-05-reserve-timeout
Merged capabilities: inventory
```

## 与场景 09 的关系

| 维度 | 场景 09 | 本场景（v0.2） |
| --- | --- | --- |
| 焦点 | 多**工具**（Cursor/Trae/Qoder/Claude）单源编译 | 多 **agent 实例** / worktree **并行试跑** |
| 隔离 | Qoder Quest 手动 `git worktree add` | `hx runtime worktree` / `--fan-out` 一等支持 |
| 人机反馈 | PR review → Steering harvest（批量、事后） | `hx review import` → 即时 fix_hints（apply 循环内） |
| 入口 | adapter sync | `--from-issue` + `hx notify` |

两者互补：09 解决「规则不漂移」，13 解决「执行怎么并行、怎么选最优、怎么把 review 注释喂回 agent」。

## 关键机制解析

- **编排不替代 Gate**：`--parallel` / `--fan-out` 只改变**怎么跑** apply；`hx verify` 的 traceability + verification 套件仍是唯一验收标准。
- **fix_hints 多源合并**：Sensor 报告 + review 标注 +（可选）Steering 蒸馏规则，统一进入 `$HX_FIX_HINTS`，agent 一次看到全部修正信号。
- **runtime.yaml vs meta.yaml**：fan-out 候选、worktree 路径是**运行态**，不写进 meta.yaml，避免污染 gate 哈希链；账本（阶段、批准、waivers）仍在 meta.yaml。
- **fan-out 记分**：verify 通过 ≈ 1000 分起，blocker 数扣分；同分取第一个。可人工 override，在合并前 review 非 selected 的 worktree diff。
- **Issue 脚手架是起点不是终点**：`--from-issue` 只生成 proposal + delta 骨架；spec 质量仍靠 gate 与人工批准（FR-012）。
