# 场景 12：开工前自定义概要设计产出模板（design.md）
| | |
| --- | --- |
| **旅程** | 定制与迁移 |
| **适用角色** | 架构师 |
| **前置** | 场景 01,11 |
| **关联** | 见 [场景选择指南](00-场景选择指南.md) |

## 背景

优惠券服务（coupon-service，场景 01 已接入 `api-service` Bundle）即将启动「批量发券」功能。架构师 **刘架构** 希望概要设计文档在默认 ADR 骨架之外，还必须包含：

- **接口清单**（对外 REST + 内部事件）；
- **数据模型变更**（表/索引）；
- **可观测性**（指标、日志字段、告警阈值）；
- **回滚策略**。

`hx design` 命令生成的 `design.md` 脚手架只有 Context / ADR / Architecture Constraints 三节。刘架构要在**第一条 design change 之前**，把「团队版 design 模板」注册进 Harness，让 agent 在 `/hx-design` 阶段按统一结构产出。

> **实现说明（当前版本）**：`proposal-template` 已由 `hx propose` 直接渲染；`design-template` 作为 `guide.template` 资产注入 design 阶段的 Context Pack，配合定制后的 `/hx-design` 命令提示词，由 agent 在脚手架基础上**按模板扩写** `design.md`。后续版本可能让 `hx design` 与 proposal 一样直接读取 `design-template` 渲染脚手架。

## 操作步骤

### 1. 新增 design 模板资产

```console
$ mkdir -p harnessX/assets/guides/design-template/examples
```

创建 `harnessX/assets/guides/design-template/template.md`：

```markdown
# Design: {{change}}

## Context

<!-- 来自 proposal.md、explore.md、现有 specs 的约束；链接相关 ADR/spec -->

## API Surface

| Method | Path | Purpose | Auth |
|--------|------|---------|------|
| | | | |

## Data Model

<!-- 新表/字段/索引；迁移策略 -->

## Decisions (ADR)

### ADR-1: <title>
- Status: proposed
- Decision:
- Alternatives considered:
- Consequences:

## Architecture Constraints

<!-- 须可被 arch-boundary / perf-budget 等传感器机械检查 -->

- Layering: routes → services → repositories（禁止反向依赖）
- ...

## Observability

- Metrics:
- Log fields (structured):
- Alerts:

## Rollback Plan

<!-- 功能开关、迁移回滚、数据修复步骤 -->
```

可选：在 `examples/bulk-issue-coupon-design.md` 放一份脱敏的「金标准」设计，供 agent 对照。

创建 `asset.yaml`（与其他 guide 资产同构）：

```yaml
id: design-template
kind: guide.template
category: architecture
phase: [design]
version: 1.0.0
origin: local
status: enforced
execution: computational
provenance: []
metrics: {}
```

### 2. 注册到 harness.yaml

在 `guides:` 列表追加（`api-design` Skill 若已通过 Bundle 存在，保留即可）：

```yaml
guides:
  - id: design-template
    kind: guide.template
    execution: computational
    phase: [design]
    source: assets/guides/design-template/template.md
  # api-service bundle 已提供：
  # - id: api-design
  #   phase: [design, apply]
```

### 3. 定制 `/hx-design` 阶段命令（关键一步）

复制默认命令为团队版，编辑 `harnessX/assets/commands/design.md`。在 **Steps** 第 2 步把「填 design.md」改为显式引用模板：

```markdown
2. Fill `harnessX/changes/<change>/design.md` using the **design-template** guide in this Context Pack:
   - Replace the minimal scaffold from `hx design` with the full section layout from the template (`API Surface`, `Data Model`, `Observability`, `Rollback Plan`, …).
   - **Context** — constraints from explore.md, proposal.md, and current specs;
   - **Decisions (ADR)** — one entry per significant decision; record rejected alternatives and WHY;
   - **Architecture Constraints** — rules arch-boundary / perf-budget sensors can enforce mechanically;
   - **API Surface / Data Model / Observability / Rollback Plan** — required for coupon-service; empty tables are not done.
```

Guardrails 可追加：

```markdown
- Every new REST endpoint listed in API Surface MUST appear in the delta spec (error responses included — see skill:api-design).
- Rollback Plan MUST mention feature flag or migration revert path when schema changes.
```

### 4. 可选：强化 api-design Skill

编辑 `harnessX/assets/bundles/api-service/skills/api-design.md`（或新建 `coupon-design` Skill 仅挂 design 阶段），补充批量发券相关的接口约定，与模板中的 **API Surface** 表格列对齐。

### 5. 自检与编译

```console
$ hx harness lint
no conflicting guide directives found

$ hx adapter sync
cursor (Tier 1): 14 file(s)
  + .cursor/commands/hx-design.md
  + .cursor/skills/...   # design-template 若被编译为 skill 目录则出现；否则主要通过 guide pack 注入
```

确认 Context Pack 已含模板：

```console
$ hx change create bulk-issue --domains coupon-issuing
$ hx propose bulk-issue --title "批量发券"
# … 填完 proposal 并通过 gate check …

$ hx guide pack bulk-issue --phase design | head -40
# Context Pack — design / bulk-issue
# …
# ## Guide: design-template (guide.template)
# … API Surface / Data Model / Observability …
```

### 6. 走 design 阶段验证

```text
Cursor ▸ /hx-design bulk-issue
```

agent 依次：

1. `hx design bulk-issue`（内部检查 proposal 完整性）；
2. 读取 Context Pack 中的 **design-template**，扩写 `design.md` 全部章节；
3. 若接口设计 imply 新 Scenario，同步更新 delta spec 并 `hx gate check --phase spec`；
4. `hx gate advance bulk-issue` → `designed`。

刘架构 review 的 `design.md` 片段示意：

```markdown
## API Surface

| Method | Path | Purpose | Auth |
|--------|------|---------|------|
| POST | /v1/coupons/bulk-issue | 创建批量发券任务 | Bearer |

## Observability

- Metrics: `coupon_bulk_issue_jobs_total{status}`
- Alerts: 失败率 > 5% 持续 10m → P2

## Rollback Plan

- Feature flag `bulk_issue_enabled` 默认 false；关闭即停写入。
- 迁移 `20260401_bulk_jobs` 可逆：down 脚本删除 `bulk_issue_jobs` 表。
```

终端推进：

```console
$ hx gate advance bulk-issue
GATE PASS (design)
advanced: proposed → designed
```

### 7. 下游如何「兑现」设计模板

| 模板章节 | 下游约束 |
|----------|----------|
| Architecture Constraints | verify 阶段 `arch-boundary`、`perf-budget` 传感器 |
| API Surface | `/hx-plan` 复核 + delta spec 中 endpoint 行为；apply 阶段 `api-design` Skill |
| ADR Consequences | `/hx-plan` 要求每条后果有对应 task |
| Observability / Rollback | 人工 review + 可选自定义 Sensor（见场景 10） |

design 阶段**没有**类似 `spec-validate` 的专用 `design-validate` 传感器——结构合规主要靠 **guide.template + 定制命令 + 人工 review**；架构违规仍由 apply/verify 阶段的 Architecture Sensors 拦截。

## 关键机制解析

- **双轨定制**：模板资产（`guide.template`）定义**长什么样**；命令资产（`guide.command`）定义** agent 必须填哪些节、何时停**；Skill（`guide.skill`）定义**领域规范**（如 RFC 7807 错误体）。
- **Context Pack 阶段隔离**：design 阶段注入 `proposal.md` + `design.md` + design 阶段 Guides，不含 apply 阶段编码 Skill 以外的噪音（见 `guideEngine.ts` 的 `PHASE_ARTIFACTS`）。
- **与 proposal 定制联用**：场景 11 在 proposal 里增加的合规字段，应被 design 模板的 **Context** 节引用，形成「需求 → 设计」可追溯链。
- **权限**：design 阶段 agent 仅可编辑 `changes/<id>/design.md` 与 `changes/<id>/specs/**`（设计驱动 spec 变更时）。

## 常见误区

1. **只改模板、不改 `commands/design.md`** — agent 可能仍按默认三步走，漏填 Observability / Rollback。命令与模板要成对改。
2. **手改 `.cursor/commands/hx-design.md`** — 下次 `adapter sync` 会被覆盖；只改 `harnessX/assets/commands/design.md`。
3. **在模板里删掉 ADR / Architecture Constraints** — 不会立刻被 Gate 拦，但 `/hx-plan` 与 verify 传感器会失去抓手；不建议删除，只扩展章节。
