Code Agent 之上的 Harness 交付系统

基于 Martin Fowler《Harness Engineering for Coding Agent Users》的控制论模型， 融合 OpenSpec 规格驱动开发（SDD）实践，设计一套约束 AI 从需求 → 设计 → 编码 → 测试 完整交付过程的 Outer Harness 系统 — HarnessX。

Harness Engineering Spec-Driven Development OpenSpec 兼容 v0.1 设计稿 2026-07-03

1. 参考框架与核心概念

1.1 Harness Engineering 控制模型

Martin Fowler 将 Coding Agent 定义为 Agent = Model + Harness。 内置 Harness（系统提示、检索、编排）之外，团队需要构建 Outer Harness， 承担两项职责：提高首次正确率（Feedforward）与提供自校正反馈（Feedback）。

🧭

Guides（前馈控制）

在 Agent 行动前引导行为：Skills、AGENTS.md、Bootstrap 脚本、架构规范、Spec 制品。

📡

Sensors（反馈控制）

在 Agent 行动后观测结果：Linter、测试、结构测试、Spec 校验、LLM-as-Judge 审查。

⚡

Computational

确定性、快速（毫秒~秒）：类型检查、单元测试、ArchUnit、OpenSpec validate。

🧠

Inferential

语义推理、较慢且非确定：AI Code Review、需求对齐审查、Mutation 质量评估。

Steering Loop：当同类问题重复出现时，人类（或 Agent）迭代改进 Harness — 将反馈信号转化为新的 Guide 或更强的 Sensor，使问题在未来发生概率降低。 这与 OpenSpec 中「实施过程中可随时回写 proposal / design / specs」的理念一致。

1.2 OpenSpec SDD 工作流映射

OpenSpec 提供轻量、无刚性阶段门的规格驱动流程。HarnessX 将其作为 Behaviour Harness 的核心制品层，并扩展 Guides/Sensors 覆盖 Maintainability 与 Architecture Fitness。

/hx:explore → /hx:propose → /hx:design → /hx:spec → /hx:apply → /hx:verify → /hx:archive
OpenSpec 概念 HarnessX 扩展 Harness 类型
specs/ 主规格库 持久化 Behaviour 真值源 + Sensor 基准 Guide + Sensor 输入
changes/<name>/proposal.md 需求意图、范围、验收标准 Feedforward Guide
changes/<name>/specs/ Delta ADDED/MODIFIED/REMOVED 需求与 Scenario Behaviour Guide + 校验 Sensor
design.md 技术方案、架构决策记录（ADR） Architecture Guide
tasks.md 可追踪实现清单 + 测试任务 Implementation Guide
/opsx:apply 带 Gate 的编码执行（必须通过 Sensor 才能推进） Runtime 编排
/opsx:archive Spec 合并 + Harness 经验沉淀 Steering Loop 输入

2. 系统架构

2.1 核心组件

组件职责关键接口
Artifact Store 管理 specs、changes、design、tasks、ADR；支持 Delta 合并与版本历史 store.read(), store.merge(), store.archive()
Guide Engine 按阶段组装 Context Pack：规格片段、Skills、AGENTS.md、拓扑模板 guide.pack(phase, changeId)
Gate Orchestrator 定义阶段转换条件；阻断未通过 Quality Gate 的推进 gate.check(phase), gate.advance()
Sensor Runner 并行/串行执行 Sensors；输出 LLM 可消费的修正指令 sensor.run(profile), sensor.report()
Steering Loop Engine 聚合失败模式；生成 Harness 补丁建议；管理 Template 版本 steer.analyze(), steer.proposePatch()
Agent Adapter 对接 Cursor / Claude Code / Codex 等；暴露 Slash Commands /hx:* 命令族

2.2 设计原则

  • Shift Quality Left — 便宜快速的 Computational Sensor 前置到每次 Agent 迭代；昂贵 Inferential Sensor 后置到 PR/CI。
  • Spec as Source of Truth — 行为需求持久化于仓库，而非聊天历史；编码与测试均 Trace 到 Spec Scenario。
  • Soft Gates, Hard Sensors — 阶段可灵活回退（OpenSpec 无 rigid phase gate），但 Sensor 失败必须阻断归档与合并。
  • Harness Coherence — Guides 与 Sensors 统一注册于 harness.yaml，避免矛盾指令。
  • Brownfield First — 支持增量接入；小改动走 Lite Profile，跳过多余制品。

3. 功能需求

3.1 交付生命周期管理

FR-001P0 系统 SHALL 支持创建 Change 工作区，包含 proposal、delta specs、design、tasks 四类制品。

FR-002P0 系统 SHALL 提供 /hx:explore 命令：只读分析代码库，输出探索笔记，不创建代码或制品（可选）。

FR-003P0 系统 SHALL 提供 /hx:propose：从自然语言意图生成 proposal.md 与初始 delta specs 草稿。

FR-004P0 系统 SHALL 提供 /hx:design：基于已批准的需求生成/更新 design.md，含 ADR 条目与架构约束。

FR-005P0 系统 SHALL 提供 /hx:spec：细化 delta specs，每个 Requirement 至少包含一个 GIVEN/WHEN/THEN Scenario。

FR-006P0 系统 SHALL 提供 /hx:plan：从 specs + design 生成 tasks.md，含实现任务与测试任务的双轨清单。

FR-007P0 系统 SHALL 提供 /hx:apply：按 tasks 顺序驱动 Agent 编码，每完成一项任务运行 Pre-Task Sensor Profile。

FR-008P0 系统 SHALL 提供 /hx:verify：运行完整 Verification Profile（Spec 覆盖、测试、结构、可选 AI Review）。

FR-009P0 系统 SHALL 提供 /hx:archive：合并 delta specs 至主 specs/，移动 change 至 archive/，触发 Steering Loop 摘要。

FR-010P1 系统 SHALL 支持 /hx:sync：检测代码与 specs 漂移，提示回写 specs 或修正代码。

3.2 Quality Gate 与 Sensor

FR-020P0 Gate Orchestrator SHALL 定义可配置阶段门：proposed → designed → specified → planned → implementing → verified → archived。

FR-021P0 每个 Gate SHALL 关联 Sensor Profile；失败时返回结构化报告，含 fix_hint 字段供 Agent 自校正。

FR-022P0 系统 SHALL 内置 OpenSpec 兼容的 delta spec 校验 Sensor（ADDED/MODIFIED/REMOVED 格式、Scenario 完整性）。

FR-023P0 系统 SHALL 支持 Spec-to-Test Traceability Sensor：每个 Scenario 至少映射一个测试用例或显式 waiver。

FR-024P1 系统 SHALL 支持注册自定义 Sensor（Shell、Node、Python 插件），并标注 computational / inferential 类型。

FR-025P1 系统 SHALL 支持 Approved Fixtures 模式：关键 Scenario 的期望输出由人类预先批准，Agent 不得修改 fixture 内容。

3.3 Guide 与 Context Pack

FR-030P0 Guide Engine SHALL 按阶段注入不同 Context Pack（见 §8.2），避免 Token 浪费与指令冲突。

FR-031P1 系统 SHALL 支持 Harness Template：按服务拓扑（API 服务、事件消费者、Dashboard）预置 Guides + Sensors _bundle。

FR-032P1 系统 SHALL 支持 Skills 注册表，与 Cursor/Codex Skills 格式兼容。

3.4 Steering Loop

FR-040P1 系统 SHALL 记录每次 Sensor 失败与人工干预，聚合为 Failure Catalog。

FR-041P1 Steering Loop SHALL 能建议 Harness 补丁：新 Skill 条目、新 ArchUnit 规则、proposal 模板改进等。

FR-042P2 系统 SHOULD 提供 Harness Coverage 报告：已注册 Controls vs 历史 Failure Mode 的覆盖矩阵。

5. 约束与边界

  • HarnessX 是 Outer Harness，不替换 Agent 内置能力，不绑定特定 LLM。
  • 不追求消除人工审查，而是将人工注意力引导至 proposal/spec 对齐与高风险决策。
  • 小改动（单文件 typo、依赖补丁）可走 profile: lite，跳过 design 与完整 verify。
  • Legacy 代码库可先只启用 Maintainability Harness，逐步补全 Behaviour Spec。
  • 制品默认存 Git，不依赖外部 SaaS（Inferential Sensor 可选云服务）。

6. 交付阶段详解

Phase 0 — Explore（可选）

输入：模糊需求、问题域。 输出：explore.md（选项分析、风险、推荐路径）。

Gates：无硬门；只读 Sensor 确认未修改代码。

Phase 1 — Propose（需求）

输出：proposal.md — Intent、Scope、Out-of-Scope、Acceptance Criteria、Stakeholders。

Gates：Proposal Completeness Sensor；可选 Inferential「需求歧义审查」。

Phase 2 — Design（设计）

输出：design.md + ADR；组件图、数据流、错误策略、可观测性设计。

Gates：Architecture Fitness Sensors（模块边界、性能预算、日志标准）。

Phase 3 — Spec（规格）

输出：Delta specs（ADDED/MODIFIED/REMOVED + Scenarios）。

Gates：OpenSpec validate；Scenario 覆盖率；与 proposal 追溯一致性。

Phase 4 — Plan（任务分解）

输出：tasks.md — 实现轨 + 测试轨；每项标注关联 Requirement ID。

Gates：每个 Requirement 至少一条 task；测试 task 不可缺失（可 waiver）。

Phase 5 — Apply（编码）

行为：Agent 逐 task 执行；每 task 后运行 Fast Sensor Profile；失败则自校正循环（上限 N 次）。

Gates：Lint、TypeCheck、相关单元测试、Spec-Task 状态同步。

Phase 6 — Verify（测试验证）

行为：全量测试、Spec Traceability、结构测试、可选 Mutation / AI Review。

Gates：所有 P0 Sensor 通过；人工 Sign-off 记录（可选）。

Phase 7 — Archive（归档）

行为：合并 Spec；归档 Change；生成 Retro 摘要；Steering Loop 输入。

与 OpenSpec 的差异：HarnessX 显式拆分 design 与 spec 阶段门， 新增 verify 阶段及三类 Harness 传感器配置。Core 模式下可合并 propose+design+spec 为单次 /hx:propose --full 以兼容 OpenSpec 快速路径。

8. Guides 与 Sensors 注册

8.1 harness.yaml 示例

version: "1.0"
profiles:
  lite:       # 小改动
    phases: [propose, apply, archive]
    sensors: [lint, unit-changed]
  standard:   # 默认
    phases: [propose, design, spec, plan, apply, verify, archive]
  strict:     # 核心域/合规
    phases: [explore, propose, design, spec, plan, apply, verify, archive]
    sensors: [..., mutation, ai-review]

guides:
  - id: coding-conventions
    type: inferential
    phase: [apply]
    source: .cursor/skills/coding-conventions/SKILL.md
  - id: api-topology
    type: feedforward
    phase: [design, apply]
    source: harnessX/templates/api-service/skills/api-design.md
  - id: proposal-template
    type: computational
    phase: [propose]
    source: harnessX/templates/proposal.md

sensors:
  - id: spec-validate
    type: computational
    phase: [spec, verify, archive]
    run: hx sensor spec-validate --change $CHANGE
    on_fail: block
    fix_hint: "Fix delta spec format; ensure each Requirement has ≥1 Scenario"
  - id: lint
    type: computational
    phase: [apply, verify]
    run: npm run lint
    on_fail: retry
    max_retries: 3
  - id: spec-trace
    type: computational
    phase: [verify]
    run: hx sensor trace --change $CHANGE
    on_fail: block
  - id: ai-spec-review
    type: inferential
    phase: [propose, verify]
    run: hx sensor ai-review --target specs
    on_fail: warn
    budget_tokens: 8000

阶段Context Pack 内容排除（防污染）

9. 三类调节域（Regulation Categories）

🛠 Maintainability Harness

调节目标：代码质量、可维护性、风格一致性。

Guides：AGENTS.md、Coding Skills、OpenRewrite 配方。

Sensors：ESLint、TypeCheck、复杂度、重复代码、覆盖率、ArchUnit。

🏗 Architecture Fitness Harness

调节目标：架构特征 — 性能、可观测性、模块边界、弹性。

Guides：性能预算 Skill、日志标准、拓扑模板。

Sensors：Perf 测试、结构测试、SLO 探针、依赖规则。

✅ Behaviour Harness

调节目标：功能行为是否符合需求。

Guides：Delta Specs、Scenario、Approved Fixtures。

Sensors：Spec validate、Traceability、集成/E2E 测试、Mutation、AI 行为审查。

关键洞察（Fowler）：Maintainability 与 Architecture 的 Computational Sensor 已较成熟； Behaviour Harness 仍是最大挑战。HarnessX 通过 OpenSpec 规格层 + Traceability + Approved Fixtures 组合， 减少对「AI 自生成测试质量」的单一依赖。

10. Steering Loop 机制

Agent 执行 → Sensor 失败/人工修正 → Failure Catalog 聚合 → 模式识别（≥3 次） → Harness Patch PR → Template 版本发布

11. 前馈与反馈资产的管理、沉淀与共享

前馈模板（proposal/design 模板、Skills、拓扑模板）与反馈规则（lint 配置、结构测试、AI Review 规则） 如果散落在各处，Harness 会迅速失去一致性（Fowler 提出的 Harness Coherence 问题）。 HarnessX 的解法是：把两者统一建模为版本化的「控制资产」（Control Asset）， 用同一套注册、分层解析、效果度量、晋级发布机制管理。

11.1 统一控制资产模型

每个 Guide 或 Sensor 都是一个带 manifest 的目录，Guides 与 Sensors 同构管理：

harnessX/assets/guides/proposal-template/
├── asset.yaml          # 元数据 manifest
├── template.md         # 模板正文（带变量占位符）
└── examples/           # 好/坏示例（供 Agent few-shot 参考）

# asset.yaml
id: proposal-template
kind: guide.template          # guide.* | sensor.* | orch.*（完整分类见下方全景表）
category: behaviour           # maintainability | architecture | behaviour
phase: [propose]
version: 1.3.0
owner: platform-team
origin: hub                   # builtin | local | team | hub
status: enforced              # draft | trial | enforced | deprecated
provenance:                   # 沉淀来源追溯
  - type: failure-pattern
    ref: FC-2026-042
metrics:                      # 效果数据（自动回填）
  first_attempt_pass_delta: +12%
  false_positive_rate: 3%
为什么同构？前馈和反馈资产共享同一套版本号、来源追溯、效果指标与发布流程， 才能实现「一条 Review 规则总是与对应的模板/Skill 成对演进」—— 例如 layering Skill（前馈）与 arch-boundary Sensor（反馈）绑定为同一 bundle，避免指令与检查相互矛盾。

控制资产全景分类

判定标准：任何能「改变 Agent 行为的概率分布」（前馈）或「产生 Agent 可消费的校正信号」（反馈） 的可版本化内容，都应纳入控制资产管理。模板与 Review 规则只是最典型的两类，完整分类如下：

Kind方向资产类型说明与示例执行类型
前馈（Guide）资产

guide.skill前馈 Skill / 知识条目 编码约定、API 设计规范、领域知识、调试指南 Inferential
guide.exemplar前馈 示例库（Few-shot） 「金标准」代码片段、优秀 PR、规范 Spec 的正反例；比抽象规则更能校准 Agent 输出 Inferential
guide.scaffold前馈 脚手架 / Bootstrap 脚本 确定性生成项目骨架、模块、测试桩；Agent 不用「猜」结构，直接调用脚本 Computational

sensor.rubric反馈 AI Review 规则 结构化 Rubric（见 §11.3），LLM Judge 逐条评估 Inferential
sensor.fixture反馈 测试资产 Approved Fixtures、golden files、契约测试、property-based 生成器 —— 人类批准的行为期望，Agent 不可篡改 Computational
sensor.budget反馈 性能预算 / SLO 探针 响应时间预算、bundle 体积上限、内存阈值 —— Architecture Fitness 的量化基准 Computational
sensor.drift反馈 漂移探测器 死代码检测、依赖扫描、spec-code 同步检查、测试质量分析 —— 变更生命周期之外持续运行 Both
sensor.mutation反馈 Mutation 测试配置 验证「测试的测试」：AI 生成的测试是否真能捕获缺陷 Computational
sensor.eval反馈 Judge 验证集 Inferential Sensor 自身的评估数据集 —— 度量 Rubric 的一致性与误报率，是「传感器的传感器」 Computational
编排（Orchestration）资产 —— 既非纯前馈也非纯反馈
orch.profile编排 Profile / Gate 定义 lite / standard / strict 的阶段与 Sensor 组合策略 —— 编排策略本身需要版本化与共享 —
orch.waiver编排 豁免策略 什么条件下允许跳过哪些 Gate、豁免的审批与过期机制 —— 防止豁免变成后门 —
orch.pattern编排 失败模式条目 Failure Catalog 中已确认的模式定义 —— 跨仓库共享后，其他团队可预防性引入对应控制 —
两个容易被忽视的类别： ① 确定性前馈（scaffold / codemod / capability）—— 把任务从「Agent 推理」降级为「脚本执行」， 是最便宜也最可靠的控制手段，应优先于写更多自然语言指令； ② 编排资产 —— Gate 组合与豁免策略决定了所有 Guides/Sensors 如何生效， 不纳入版本管理会导致「同一套规则在不同仓库执行强度不一致」的隐性漂移。

11.2 前馈资产：模板的管理

模板三层结构

  • 制品模板 — proposal / design / spec / tasks 的骨架，含变量占位符与「填写指引注释」（Agent 可读的 inline instruction）。
  • Skill 模板 — 编码约定、API 设计规范等知识型 Guide，兼容 Cursor / Claude Skills 格式。
  • 拓扑 Harness 模板 — 按服务类型打包的 Guides + Sensors bundle（如 api-service），对应 Fowler 的 Harness Template 概念。

分层解析（Layered Resolution）

同一资产 ID 按优先级从高到低解析，高层可覆盖低层，覆盖必须显式声明：

Change 级覆盖 > 仓库本地 assets/ > 团队模板包 > 组织 Harness Hub > HarnessX 内置 
# harness.yaml 中声明依赖与覆盖
dependencies:
  - "@org/api-service-harness@^2.1"    # 来自 Hub 的 bundle
overrides:
  - id: proposal-template
    source: harnessX/assets/guides/proposal-template   # 本地定制版
    reason: "增加合规审批段落（金融域要求）"            # 覆盖必须写明原因

hx guide pack 在组装 Context Pack 时完成解析与渲染； harness.lock 锁定所有资产的精确版本，保证团队成员与 CI 使用一致的 Harness。

11.3 反馈资产：Review 规则的管理

Computational 规则

lint 配置、结构测试、自定义 Sensor 脚本直接作为 sensor.rule / sensor.script 资产管理， 封装要点是输出必须转换为 LLM-Optimized 报告格式（含 fix_hint，见 §14.3）。

Inferential 规则：Rubric as Data

AI Review 规则不写成自由文本 prompt，而是结构化 Rubric —— 每条规则独立编号、可单独启停、可单独度量误报率： 

# harnessX/assets/sensors/api-review-rubric/rubric.yaml
id: api-review-rubric
model_hint: high-reasoning        # 建议的 judge 模型档位
rules:
  - id: R-API-001
    check: "所有对外 endpoint 是否有显式错误响应定义（4xx/5xx schema）"
    severity: block               # block | warn | info
    scope: ["src/api/**"]
    good_example: |
      @ApiResponse(status = 400, schema = ErrorResponse.class)
    bad_example: |
      // endpoint 只声明了 200 响应
    fix_hint: "为每个 endpoint 补充 ErrorResponse schema 定义"
  - id: R-API-002
    check: "分页参数是否遵循 cursor-based 规范而非 offset"
    severity: warn
    scope: ["src/api/**"]
    fix_hint: "参考 skill:api-design §分页 改为 cursor 分页"

LLM Judge 逐条评估 Rubric 并输出结构化 findings（与 Computational Sensor 报告同格式），使 Gate Orchestrator 可统一处理。

规则生命周期

draft（草稿） → trial（试运行，仅 warn） → enforced（可 block） → deprecated（退役）
  • trial 期：规则只产生 warning，系统统计触发率与人工标注的误报率；误报率 < 10% 且触发有效才可晋级 enforced。
  • Inferential 规则默认最高 warn：非确定性规则不建议作为硬 block（见 §18.3 开放问题），除非有高一致性验证集背书。
  • 退役条件：连续 N 个周期零触发且对应风险已被上游 Guide 消除，或误报率持续超标。

11.4 沉淀机制：从经验到资产

沉淀的三个来源，统一汇入 Steering Loop 的蒸馏流水线：

📉

Sensor 失败模式

Failure Catalog 中重复 ≥3 次的模式，自动生成资产补丁草稿（新 Skill 条目 / 新 Rubric 规则 / lint 规则脚手架）。

💬

人工 Review 意见

hx steer harvest-pr 收集 PR 中人类 reviewer 的修改意见，聚类后转化为 Rubric 规则草稿 —— 人的判断被逐步外化为可复用规则。

📋

Archive Retro

每次 /hx:archive 生成回顾摘要：本次哪些环节返工最多、哪些 Guide 缺失，作为蒸馏输入。

蒸馏流水线（hx steer distill）

聚合三类信号 → 模式聚类 → 生成 draft 资产（Agent 辅助） → 人工审核 PR → trial 试运行 → enforced
  • 证据链：每个蒸馏出的资产在 provenance 中链接原始 failure / PR comment，审核者可追溯「这条规则为什么存在」。
  • 数据驱动晋级：资产的 metrics（首次通过率提升、误报率、触发率）由运行遥测自动回填，晋级与退役都以数据为准，避免规则只增不减导致 Harness 膨胀。
  • 成对沉淀：蒸馏时优先「前馈 + 反馈成对生成」—— 例如同时产出 layering Skill 条目（教 Agent 怎么做）和 arch-boundary 规则（检查是否做对），单独的 Sensor 只能反复报错，单独的 Guide 无法验证是否生效。

11.5 共享机制：Harness Hub

组织级共享通过一个中心 Git 仓库 harness-hub 实现（类比私有 npm registry，但资产即目录、Git 即分发）： 

harness-hub/
├── packages/
│   ├── api-service-harness/        # 拓扑级 bundle
│   │   ├── package.yaml            # 版本、依赖、CHANGELOG
│   │   ├── guides/
│   │   └── sensors/
│   ├── frontend-harness/
│   └── common-review-rubrics/      # 纯规则包（跨拓扑通用）
├── evals/                          # 资产质量验证
│   ├── golden-repos/               # Sensor 在标准仓库上的自测基准
│   └── rubric-eval-sets/           # Inferential 规则的一致性验证集
└── .github/workflows/validate.yml  # 资产 CI：schema 校验 + 自测

消费端

hx hub add @org/api-service-harness@^2     # 添加依赖，写入 harness.yaml + lock
hx hub sync                                # 拉取上游更新，输出 diff 报告
hx hub sync --apply                        # 三方合并：上游新版 vs 本地 override vs 基线

贡献端（沉淀 → 共享的晋级路径）

仓库本地沉淀 → 团队内验证（metrics 达标） → hx hub promote <asset> → Hub PR（附证据链+指标） → Hub CI 验证 → SemVer 发布
  • Hub CI 把关：Computational Sensor 必须在 golden repos 上通过自测（能查出植入的坏例、不误报好例）；Inferential Rubric 必须在验证集上达到一致性阈值（同一输入多次评估结论稳定）。
  • 解决 Template Drift（Fowler 指出的核心难题）：资产按引用 + lock 消费而非复制粘贴；本地定制必须走显式 overrides 声明；hub sync 用三方对比合并上游改进与本地定制，而不是二选一。
  • 跨仓库信号聚合：Hub 定期收集各消费仓库上报的资产 metrics（匿名化），当同一失败模式出现在多个仓库时，优先在 Hub 层修复而非各仓库各自打补丁。
小结：管理 = 统一资产模型 + 分层解析 + lock 版本化； 沉淀 = 三源信号 → 蒸馏流水线 → 证据链 + 数据驱动的生命周期晋级； 共享 = Harness Hub 的「引用消费 / 验证发布 / 三方同步」。 前馈与反馈成对演进，是保持 Harness Coherence 的根本手段。

14. 运行时引擎设计

14.1 Gate 状态机

type Phase =
  | "explore" | "proposed" | "designed" | "specified"
  | "planned" | "implementing" | "verified" | "archived";

interface GateResult {
  passed: boolean;
  sensorResults: SensorResult[];
  blockers: string[];      // 必须修复
  warnings: string[];      // 可继续但需记录
  fixHints: string[];      // 聚合给 Agent 的修正指令
}

// 阶段推进规则：仅当 blockers 为空
function advance(change: Change): Phase | null

14.2 Apply 循环（核心）

async function applyLoop(changeId: string) {
  const tasks = loadTasks(changeId).filter(t => !t.done);
  for (const task of tasks) {
    const pack = guideEngine.pack("apply", changeId, { currentTask: task });
    await agent.run(pack, task.description);

    let retries = 0;
    while (retries < MAX_RETRIES) {
      const result = await sensorRunner.run("fast", changeId);
      if (result.passed) break;
      await agent.run(result.fixHints, "Fix sensor failures");
      retries++;
    }
    if (retries >= MAX_RETRIES) throw new GateBlockedError(task.id);
    markTaskDone(task);
    updateTraceability(changeId, task);
  }
}

17. 度量与 Harness 质量评估

Fowler 提出：需要类似 Code Coverage 的「Harness Coverage」概念。HarnessX 建议跟踪：

指标定义目标
First-Attempt Pass Rate Apply 阶段 task 首次 Sensor 通过率 > 70%（成熟 harness）
Spec Traceability Coverage 有测试映射的 Scenario 占比 100% P0 requirements
Gate Rework Count Change 在各阶段被退回次数 下降趋势
Failure Pattern Recurrence 相同 Failure Catalog 条目重复率 Steering 后 < 10%
Human Review Minutes 人工审查 proposal/spec 时长 相对无 harness 基线下降 40%+
Token Efficiency 完成 change 的总 token / 成功归档 change 数 随 harness 成熟下降
