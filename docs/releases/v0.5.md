# HarnessX v0.5

**Tag:** `v0.5` · **npm version:** `0.5.0` · **Node:** ≥ 20

HarnessX v0.5 introduces the **four-stage delivery model** (`req` → `arch` → `dev` → `test`), aligning org-level requirements/architecture with change-level development and formal testing — while keeping `delivery_mode: phases` as the default for v0.4 compatibility.

---

## Highlights

### Four-stage delivery (`req` / `arch` / `dev` / `test`)

Authoritative registry in `packages/core/src/stages.ts`. Each stage has tasks with required/optional flags matching the AI delivery phase specification.

| Stage | Output | Scope |
| --- | --- | --- |
| `req` | Product requirements document | Org `docs/prd/` |
| `arch` | HLD + module LLD | Org `docs/architecture/` |
| `dev` | Runnable code | Change pipeline |
| `test` | Test cases + test report | Change `test-cases/` |

Docs: [delivery-stages.zh-CN.md](../delivery-stages.zh-CN.md)

### Stages gate engine (opt-in)

```yaml
# harnessX/config.yaml
delivery_mode: stages
```

- `stageGateCheck` / `stageAdvance` / `nextTask`
- `meta.yaml` fields: `stage`, `task`, `stageProgress`, `taskHistory`
- Approval gate: `design-to-plan` (legacy `spec` still accepted)

### New CLI commands

```bash
hx req prd init <slug> --title "..."
hx dev status <change>
hx test status <change>
hx stage status [--stage req|arch|dev|test]
hx migrate [--dry-run] [--delivery-mode stages]
hx gate check <change> --stage dev --task propose
```

Legacy commands (`hx prd`, `hx propose`, `hx gate check --phase`) remain as aliases.

### Profile extensions

`harness.yaml` profiles now support `stages`, `dev_tasks`, `test_tasks`, and `dev.*` / `test.*` suite keys alongside legacy `phases`.

### Fixes bundled in v0.5

- Gate `ORDER` includes `test_designed` (fixes SDLC profile advance)
- `test-design` support in guideEngine and L1 contract (`HX_STAGE`, `HX_TASK`)
- `enterprise-sdlc` hub blueprint phase list corrected

---

## Upgrade from v0.4

```bash
hx migrate --dry-run
hx migrate                    # writes stage/task to meta.yaml
hx migrate --delivery-mode stages   # opt into stages gate engine
```

Migration guidance is preserved in this release note; current projects should use the four-stage manuals linked from the repository README.

---

## Breaking changes

None by default (`delivery_mode: phases`). Opting into `delivery_mode: stages` changes gate CLI parameters and meta shape; use `hx migrate` first.
