# HarnessX / hx-lite v0.5

**Tag:** `v0.5` · **npm version:** `0.5.0` · **Node:** ≥ 20

hx-lite v0.5 ships the **four-stage delivery model** (`req` → `arch` → `dev` → `test`) plus a simplified **Sensor configuration surface** (`inline` / `shell` / `rules`).

---

## Highlights

### Four-stage delivery (`req` / `arch` / `dev` / `test`)

Authoritative registry in `packages/core/src/stages.ts`.

| Stage | Output | Scope |
| --- | --- | --- |
| `req` | Product requirements document | Org `docs/prd/` |
| `arch` | HLD + module LLD | Org `docs/architecture/` |
| `dev` | Runnable code | Change pipeline |
| `test` | Test cases + test report | Change `test-cases/` |

Docs: [delivery-stages.zh-CN.md](../delivery-stages.zh-CN.md)

### Sensor three-kind config (`inline` / `shell` / `rules`)

Feedback checks in `harness.yaml` use a single public field `check`:

| `check` | Meaning | Key fields |
| --- | --- | --- |
| `inline` | Built-in predicates or `handler.<id>` | `expr` |
| `shell` | External command | `run`, `output`, `HX_*` env |
| `rules` | Rules text + LLM/heuristic judge | `rules_text` / `rules_file`, `input` |

Handbook: [sensor-config-manual.zh-CN.md](../sensor-config-manual.zh-CN.md)

### Stages gate engine

- `stageGateCheck` / `stageAdvance` / `nextTask`
- `meta.yaml` fields: `stage`, `task`, `stageProgress`, `taskHistory`
- Stage `init` creates directories (not doc templates); authoring via commands/skills

### CLI / TUI

```bash
hx req prd init <slug>
hx stage status [--stage req|arch|dev|test]
hx gate check <change> --stage dev --task propose
hx tui [--locale zh]
```

## Breaking changes

- Sensor YAML no longer uses `builtin` / `engine` / `impl` / `plugin` as the public API; migrate to `check` + `expr` / `run` / `rules_*`.
- Complex TS checks use `expr: "handler.<id>"`.
